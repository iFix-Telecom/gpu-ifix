<objective>
Fix the TTS health prober so it probes each tts-role upstream with the request
shape that upstream actually serves.

The live request path is already correct (gateway/internal/proxy/tts.go's
piperTTSAdapter translates OpenAI POST /v1/audio/speech into Piper POST /tts
{"text":...}). ONLY the prober is wrong: probe.go's `dispatch` `case "tts":`
hardcodes `POST <url>/v1/audio/speech` with an OpenAI body for EVERY tts
upstream. The tier-1 `voice-api-piper` upstream (URL_ENV UPSTREAM_TTS_PIPER_URL)
only serves `POST /tts` with `{"text":"..."}`, so its probe gets a 404 and is
classified `config` instead of `ok`.

Purpose: stop the false `config` status on a healthy Piper fallback so ops
dashboards reflect real upstream health.
Output: per-upstream tts probe dispatch + a regression test.
</objective>
