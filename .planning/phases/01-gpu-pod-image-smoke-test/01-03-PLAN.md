```dockerfile
# syntax=docker/dockerfile:1.7

# =========================================================================
# Stage 1: llama-bin — extract the llama-server binary from the upstream
#                     CUDA image. Keeps our final image small (D-01).
# =========================================================================
FROM ghcr.io/ggml-org/llama.cpp:server-cuda AS llama-bin

# (nothing to do; we only COPY from this stage in the final stage)

# =========================================================================
# Stage 2: runtime — CUDA 12.4 runtime + curl/ca-certs for weight download
#                    and health probes. No Python, no pip.
# =========================================================================
FROM nvidia/cuda:12.4.1-cudnn-runtime-ubuntu22.04 AS runtime

# Ifix convention: TZ + locale + minimal apt surface
ENV DEBIAN_FRONTEND=noninteractive \
    TZ=America/Sao_Paulo \
    LANG=C.UTF-8 \
    LC_ALL=C.UTF-8 \
    PYTHONUNBUFFERED=1

RUN apt-get update \
  && apt-get install -y --no-install-recommends \
      ca-certificates \
      curl \
      tzdata \
      tini \
  && ln -fs /usr/share/zoneinfo/America/Sao_Paulo /etc/localtime \
  && dpkg-reconfigure -f noninteractive tzdata \
  && rm -rf /var/lib/apt/lists/*

# Copy llama-server binary from upstream CUDA image
COPY --from=llama-bin /app/llama-server /usr/local/bin/llama-server
# Copy shared libs the llama-server binary needs (ggml runtimes, BLAS, etc.)
COPY --from=llama-bin /app/libggml*.so /usr/local/lib/
COPY --from=llama-bin /app/libllama*.so /usr/local/lib/
RUN ldconfig

# Bundle the patched Jinja template into the image (from plan 02)
COPY pod/templates/qwen3.5-27b-tool-calling.jinja /app/templates/qwen3.5-27b-tool-calling.jinja
COPY pod/templates/qwen3.5-27b-tool-calling.jinja.sha256 /app/templates/qwen3.5-27b-tool-calling.jinja.sha256

# Sanity: verify template integrity at build time (fails the build if drift)
RUN EXPECTED="$(tr -d '[:space:]' < /app/templates/qwen3.5-27b-tool-calling.jinja.sha256)" \
  && ACTUAL="$(sha256sum /app/templates/qwen3.5-27b-tool-calling.jinja | awk '{print $1}')" \
  && test "$EXPECTED" = "$ACTUAL" || (echo "FATAL: template SHA-256 drift — expected $EXPECTED got $ACTUAL"; exit 1)

# Prepare weights mountpoint (weights bind-mounted at runtime, not copied)
RUN mkdir -p /weights/qwen /weights/whisper /weights/bge-m3 /app/templates

EXPOSE 8000
STOPSIGNAL SIGTERM
ENTRYPOINT ["/usr/bin/tini", "--"]
# Default CMD is the llama-server invocation (D-07, D-08, POD-06)
CMD ["llama-server", \
     "--host", "0.0.0.0", \
     "--port", "8000", \
     "-m", "/weights/qwen/model.gguf", \
     "-np", "2", \
     "--ctx-size", "16384", \
     "--jinja", \
     "--chat-template-file", "/app/templates/qwen3.5-27b-tool-calling.jinja"]
```

Key design decisions baked into the above:
- Service names `llama`, `speaches`, `infinity`, `health-bridge`, `dcgm-exporter` — match PATTERNS.md §Divergences and plan 04 interface contract (health-bridge dials `http://llama:8000` etc).
- `x-gpu-all` anchor reserves one GPU per service that needs one (llama, speaches, infinity, dcgm-exporter). health-bridge needs no GPU.
- Healthcheck start_period 120s on inference services: first-token latency after cold load can be 60-90s; 120s avoids false RESTARTs (PITFALLS §Pitfall 1 — VRAM collapse causes OOM but NOT cold-start failure).
- `stop_grace_period: 30s` on inference services (shutdown window for CUDA context cleanup).
- Bind-mount `/weights` read-only into llama, read-write into speaches (Speaches caches whisper weights on first boot) and infinity (HF cache).
- `depends_on` only for health-bridge — Compose v3+ `depends_on` waits for start, not healthy, unless `condition: service_healthy` added. Intentionally omit health condition so health-bridge starts first and reports "degraded" until upstreams become healthy — matches D-11 probe model (probes run every 10s and report real state).
- No Traefik labels (pod doesn't terminate TLS — Vast.ai port-forwards directly per PATTERNS.md divergences).
- `name: ifix-ai-pod` at top — makes `docker compose` commands use this project name regardless of cwd.
- All services on custom bridge network `ifix-ai-pod` — DNS-based discovery (container name = hostname).
- Environment variables `LOG_LEVEL`, `ENV`, `IFIX_AI_POD_IMAGE`, `IFIX_AI_POD_HEALTH_BRIDGE_IMAGE`, `WEIGHTS_DIR` interpolated from onstart.sh / Portainer UI (matches PATTERNS.md).
