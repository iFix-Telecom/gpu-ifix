<voice_persistence_contract>
## Voice persistence contract (REVISED for Chatterbox zero-shot — REVIEWS action #1, D-08 revised)

Vast.ai pod storage is EPHEMERAL. Chatterbox clones ZERO-SHOT from a reference WAV — there is NO `.pt` speaker embedding to persist or regenerate. The single source of truth is the S3 reference WAV:

```
voice_id  ─┬─►  Postgres ai_gateway.voices row   (id=voice_id, tenant_id, label, s3_key)   [DURABLE — Plan 04/07]
           └─►  MinIO S3 object  s3_key = <S3_VOICE_PREFIX>/<voice_id>.wav  (reference WAV)  [DURABLE — written by Plan 07 upload]

          ┌─►  pod-local WAV cache  <CHATTERBOX_VOICE_CACHE_DIR>/<voice_id>.wav  (OPTIONAL latency cache)  [EPHEMERAL — rebuildable]
```

Resolution mechanism = LAZY (no embedding step, just a WAV fetch):
- The gateway upload (Plan 07) persists ONLY the reference WAV to S3 + writes the catalog row. It does NOT push any `.pt`.
- On a `/v1/audio/speech` request with a `voice_id`:
  1. if a local cached WAV exists (`<CHATTERBOX_VOICE_CACHE_DIR>/<voice_id>.wav`), use it as `audio_prompt_path`;
  2. else fetch `<S3_VOICE_PREFIX>/<voice_id>.wav` from MinIO (creds from env), optionally cache it locally, and pass it as `audio_prompt_path`;
  3. if the S3 object does not exist for that `voice_id` → 404 (unknown voice);
  4. then call `m.generate(text, language_id="pt", audio_prompt_path=<wav>, ...)`.
- `voice=="random"` or no voice → `audio_prompt_path=None` (default voice).

This makes a voice survive pod replacement: cache miss on a fresh pod → refetch the reference WAV from S3 → pass as `audio_prompt_path` → synth. There is NO embedding regeneration. `test_voice_cache_miss_refetches_from_s3` asserts exactly this (S3 refetch path, no `.pt`).
</voice_persistence_contract>
