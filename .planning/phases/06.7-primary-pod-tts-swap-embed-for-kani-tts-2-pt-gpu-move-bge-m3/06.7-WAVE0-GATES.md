## GATE 3 — Piper fallback contract → **Option A: gateway adapter** (locked)

**Live Piper confirmed** (`voice-api-piper-1` on vps-ifix-vm, model `pt_BR-miro-high.onnx`):
- `POST /tts {text, voice?}` → **raw mu-law (ulaw) 8kHz, Content-Type `audio/basic`**, headers `X-Duration-Ms`/`X-Generation-Ms`. Round-trip verified: HTTP 200, 16440 bytes for "Olá, seu boleto venceu ontem." (2055ms audio, 7546ms gen, CPU RTF ~3.7).
- `GET/POST /health` → 200 `{status:ok}`. Missing `text` → 400. `voice` resolves `/models/{voice}.onnx` (path-traversal sanitized), falls back to default.

**Locked contract = Option A (gateway adapter).** Plan 07 implements verbatim:
1. **Conversion target:** WAV 16kHz 16-bit PCM mono (stable contract for off-peak clients; NOT raw telephony ulaw).
2. **Conversion tool:** pure-Go mu-law decode (256-entry LUT) + RIFF/WAV header writer — **no ffmpeg dependency in the gateway image**.
3. **JSON→form translation:** `/v1/audio/speech {input, voice}` → Piper `POST /tts {text: input, voice: voice}`.
4. **Content-Type:** set `audio/wav` on the converted response (`audio/pcm` raw PCM16LE if `response_format=pcm`).
5. **response_format:** support `wav` (default) + `pcm`; unsupported (`mp3`/`opus`/`flac`/`aac`) → clean OpenAI-shaped 400.

Note: primary (Chatterbox) emits 24kHz WAV; tier-1 Piper fallback emits the adapter's 16kHz WAV. Both are valid WAV — clients handle sample-rate via WAV header.
