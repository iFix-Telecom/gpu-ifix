## Decisions Made

- **ListPrimaryLifecycles query was missing.** The plan body in Task 1 instructed "SELECT FROM primary_lifecycles ORDER BY started_at DESC LIMIT N" but Plan 06.6-07 shipped only the 6 lifecycle-mutation queries + GetOpenPrimaryLifecycle (no ListPrimaryLifecycles). Adding the SQL + running sqlc generate produced the necessary `gen.ListPrimaryLifecycles` binding. Treated as Rule 3 (blocking issue) — auto-fixed in Task 1 commit because the entire `runPrimaryLifecycles` subcommand body depends on it.
- **DrainStartedAt is exposed in ListPrimaryLifecyclesRow.** The emergency_lifecycles analog excludes a drain column because emerg has no drain phase (it's a hard-cutover FSM). Primary lifecycles have a deliberate ramp-down window (PRIMARY_POD_SCHEDULE_GRACE_RAMP_DOWN_SECONDS) audited via drain_started_at. The tabwriter output column "DRAIN" sits between STARTED and ENDED so the operator sees the 3-phase lifecycle at-a-glance.
- **runPrimarySchedule is a PURE FUNCTION.** No Redis or Postgres I/O. config.Load() runs in the env-driven wrapper, but the testable inner body `runPrimaryScheduleWithCfg(ctx, cfg, log)` accepts an already-built cfg so unit tests can pin specific PrimaryPodSchedule* values without env mutation. This enables `TestRunPrimarySchedule_ShowsProvisionLeadSeconds` (reviews #8) to assert against an in-memory cfg built by `defaultPrimaryCfg()`.
- **ReplicaID = "gatewayctl-cli" (not "gatewayctl").** Emerg uses bare "gatewayctl" — primary disambiguates the publisher so a downstream observer correlating force_up_request → handleForceUpRequest can grep audit logs for "gatewayctl-cli" and see ONLY primary CLI activity (not emerg). Matches the plan task body literal.
- **Default --limit for primary lifecycles is 20.** Emerg defaults to 50 because incidents can be bursty. Primary lifecycles are at most 1 per scheduled day under normal operation (8h-22h window = single lifecycle), so 20 covers ~3 weeks of normal history. Operator can override via `--limit 100` for monthly review.
- **Capture helpers renamed (capturePrimaryStdout / capturePrimaryStderr).** Even though emerg_test.go has `captureStdout` / `captureStderr` with identical bodies, the rename prevents the two test files from accidentally coupling via package-level symbols. If a refactor in one file changes its helper, the other is insulated.
- **Tests committed in Task 2 against pre-existing Task 1 implementation (not strict TDD RED → GREEN).** The plan marked Task 2 `tdd="true"` but Task 1 had already shipped working code. Compressed RED/GREEN into the test commit which validates every contract claim. Each test asserts a specific behavior that would fail if the corresponding impl path regressed.
