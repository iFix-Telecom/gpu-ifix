## Decisions Made

- **Partial completion is the locked pattern for blocking-human plans.**
  Tasks 1+2 ship docs autonomously; Task 3 stays deferred. STATE.md will
  note `06.6-11 is autonomous:false HUMAN-UAT — Tasks 1+2 done…Task 3 is a
  blocking human-verify checkpoint` (Phase 6.5-11 parity). The executor
  does NOT spend Vast.ai credit; the operator does.

- **Scenario 5 drain assertion uses DB events JSONB as load-bearing surface,
  not container log grep.** The reconciler does NOT emit a textual
  `transition ready→draining reason=schedule_window_exited` slog line — it
  emits structured events via the FSM onChange callback (publishes Redis
  events with `Type="transition"` + `State="draining"` +
  `Reason="schedule_window_exited"`) AND via `startDrain` publishes
  `Type="draining_started"` + `Reason="schedule_window_exited"` AND
  appends to the DB events JSONB column. The UAT operative assertion uses
  the DB column because it survives Redis restarts. The container log
  line `"primary drain complete; transitioning to Destroying"` (slog.Info
  from `evaluateDraining`) is captured as a secondary cross-check on
  ASSERTION 2 (Draining→Destroying transition).

- **Scenario 1 ASSERTION 2 supervisord probe is conditional.** Only when
  `POD_DEBUG_SSH_PUBLIC_KEY` is set (dev mode) does the operator run
  `ssh root@<pod-ip> supervisorctl status`. In production
  (`POD_DEBUG_SSH_PUBLIC_KEY=""` per D-12 least-privilege), the assertion
  degrades to the 4-endpoint same-IP-different-port reachability test
  (ASSERTION 1) — which already proves single-container colocation but
  does not directly observe supervisord's process management. Acceptable
  trade-off: production safety > observability convenience.

- **Image build pre-flight is the third Wave 0 closure check.** Without
  `ghcr.io/ifixtelecom/converseai-primary-pod:dev-{sha}` published to
  GHCR, Scenarios 1/3/4 would not work end-to-end (the b9191 upstream-
  direct fallback path doesn't bundle speaches/infinity/dcgm). The UAT
  Prerequisites pre-flight section explicitly verifies the image exists
  via `docker pull` OR `gh run list --workflow=build-primary-pod.yml`.

- **Image bump policy is a published RUNBOOK section (not just a Wave 0
  GATES decision).** Operator can run the bump end-to-end without re-
  reading the Wave 0 planning docs each time. Reduces friction for routine
  upstream version drift while preserving the audit trail requirement
  (RUNBOOK explicitly links bumps back to WAVE0-GATES.md amendment +
  operator sign-off).

- **Cost budget for UAT is R$10-15 (~$2-3 USD).** Higher than Phase 6.5-11
  ~R$10-15 because Scenario 6 adds 2 cold-starts (emerg ~10min + primary
  ~30min) + 30-40min overlap window of dual-pod billing. This overlap is
  EXPECTED — it's the whole point of testing Pitfall #11.
