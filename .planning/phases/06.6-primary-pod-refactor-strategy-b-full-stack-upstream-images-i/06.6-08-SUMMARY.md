## Decisions Made

- **Adapter interface satisfaction via duck typing.** The concrete `*upstreams.Loader`, `*dcgm.Scraper`, and `*shed.InflightRegistry` types all satisfy `primary.LoaderAdapter` / `primary.DCGMScraperAdapter` / `primary.InflightAdapter` respectively because the method signatures match exactly (verified by 06.6-06b plan). No type-asserts or wrapper types required — Go's implicit interface satisfaction does the work. The plan's `interfaces` block in the YAML frontmatter explicitly notes this.
- **FSM `stateChangeWriter` left nil.** `audit.Writer.WriteStateChange(kind string, ev Event)` has a different signature than `primary.stateChangeWriter.WriteStateChange(kind string, ev any) error` (different parameter type AND different return). The plan body originally wrote `primary.NewFSM(auditWriter, nil)` but that does not compile; auditing via DB is deferred to a future plan. Inline comment documents the deferral. The FSM tolerates nil writer (fsm.go:137-140) — transitions still update the atomic state, only the audit hook is skipped.
- **Pitfall #11 wiring is conditional on `emergReconciler != nil`.** When `VAST_AI_API_KEY` is empty, BOTH emerg AND primary stay unconstructed (they share the same key). The `if emergReconciler != nil` gate ensures `emergReconciler.SubscribePrimaryEvents` is never called on a nil receiver. This matches the existing emerg block's "skip if no key" semantics.
- **Reviews #2 implementation:** `go primaryReconciler.Start(ctx)` is called UNCONDITIONALLY (no DISABLED gating at main.go layer). The DISABLED kill-switch is enforced inside `runScheduleLoop`'s tick-decision point (primary/reconciler.go:193). This preserves the contract that the event subscriber goroutine (handling `force_up_request` / `force_down_request`) always runs — `gatewayctl primary force-up` works under the WAVE0-GATES Decision 5 soak-gate posture.
- **Boot log surfaces `template_image` (Wave 0 SHA pin) + `provision_lead_seconds` (reviews #8).** When operators audit a failing primary lifecycle, the boot log immediately reveals which image was supposed to run, which directly answers the "is this the expected Wave 0 LOCKED build?" forensics question.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] FSM writer is nil instead of `auditWriter`**

- **Found during:** Task 2 (wiring main.go)
- **Issue:** The plan's `interfaces` block specified `primaryFSM := primary.NewFSM(auditWriter, nil)`, but `audit.Writer.WriteStateChange(kind string, ev Event)` does not satisfy the unexported `primary.stateChangeWriter` interface `WriteStateChange(kind string, ev any) error` (different parameter type AND different return type). Compiling the plan's literal expression would have failed with "cannot use auditWriter (variable of type *audit.Writer) as primary.stateChangeWriter".
- **Fix:** Passed `nil` for the writer argument (the FSM tolerates nil — fsm.go:137-140 explicitly documents nil-writer semantics) and added an inline comment in main.go documenting the signature mismatch + deferral. The onChange callback (the second argument) is fully implemented as planned and mirrors transitions to Redis. No data loss — the gw:primary:state Hash + gw:primary:events Pub/Sub paths are intact; only the DB-side audit_log emission for primary FSM is deferred.
- **Files modified:** `gateway/cmd/gateway/main.go`
- **Verification:** `go build ./...` clean; the existing `emergFSM.SetAuditWriter(auditWriter)` pattern (lines 706-710 of main.go) shows the EmergFSM has a typed setter that does the type matching, but primary FSM does not have one — adding it is out of scope for this plan.
- **Committed in:** `bda05fb` (Task 2 commit)
- **Future remediation:** A future plan can either (a) extend `primary.FSM` with `SetAuditWriter(*audit.Writer)` that wraps the typed call in an `any`-shaped closure, or (b) modify `audit.Writer.WriteStateChange` to return `error` and accept `any`. Option (a) is the smaller change and parallels emerg's `SetAuditWriter` pattern.

## Threat Model Compliance

**T-06.6-04 (DoS via emerg+primary double-provision race / Pitfall #11)** — mitigated. The Option B primary-precedence handoff is now wired end-to-end:
1. Primary reconciler reaches `StateReady` and publishes `primary_ready` event on `gw:primary:events`.
2. Emerg reconciler's `SubscribePrimaryEvents` goroutine (leader-only) observes the event.
3. If emerg FSM is `StateEmergencyActive`, `cancelActiveLifecycle("primary_took_over")` fires — Layer 1 (ctx cancel of provisioning goroutine), Layer 2 (cross-replica `cancel_in_flight` publish), Layer 3 (post-create destroy enforced inside `waitForReadyOrDestroy`).
4. Net effect: single tier-0 LLM pod active at steady-state during peak windows; graceful handoff during transition.

The 3 unit tests in `subscribe_test.go` cover the happy path (leader + EmergencyActive → cancel) + 2 negative paths (non-leader, FSM ≠ EmergencyActive → no-op), giving regression coverage for the threat mitigation.

No new threat surfaces introduced — `gw:primary:events` Pub/Sub was already publishable (Plan 06.6-07 + 06.6-06a) and the existing redisx.SubscribePrimaryEvents was already declared. This plan only adds a NEW emerg-side consumer.

## Threat Flags

None. The new boot-log fields (`template_image`, `provision_lead_seconds`, etc.) are configuration knobs already validated by `cfg.Load`; surfacing them in logs is a forensics-positive change without introducing a new external surface.
