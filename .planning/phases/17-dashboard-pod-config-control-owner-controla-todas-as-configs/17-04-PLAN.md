<objective>
Build the three gateway admin endpoints the dashboard consumes:
1. `GET /admin/primary/lifecycle` (read) — current FSM state + the OPEN lifecycle event trail (D-05),
   mirroring `admin.OperationsHandler` (PATTERNS.md §lifecycle.go). Powers the live-status panel poll.
2. `GET /admin/primary/config` (read) — the CURRENT pod_config row (16 hot fields + 9 bound pairs)
   read via the `GetPodConfig` sqlc query (Plan 17-01). This is the read seam the dashboard editor uses
   for current values, the server action uses to refetch the LIVE bound during validation, and the audit
   uses to source `oldValue`. It reads `pod_config` — NOT `/admin/operations`, whose `r.cfg` is the boot
   env and DIVERGES from pod_config after any edit.
3. `PATCH /admin/primary/config` (write) — the gateway-side write seam (the ONE net-new mutation verb;
   PATTERNS.md §No-Analog) that runs `UpdatePodConfigField` / `UpdatePodConfigBound` → trigger →
   loader reload. The dashboard owner server action (Plan 17-05) calls this with X-Admin-Key.

All three mount inside `adminRouter` (X-Admin-Key gated). NO restart endpoint, NO structural edit (D-01/D-02).

Purpose: expose the live-diagnostic panel, the config READ that backs the editor/validation/audit, and the
write path that makes dashboard edits land in pod_config.
Output: three admin handlers (two read + one write) + tests + main.go mounts.

## Contract (consumed by Plan 17-05 — keep field-for-field stable)
- `GET /admin/primary/lifecycle` → 200 JSON: `{ fsm_state, leader, emergency_state, open_lifecycle: { id, trigger_reason, started_at, first_health_pass_at, drain_started_at, ended_at, accepted_dph, total_cost_brl, shutdown_reason, events[] } | null }`. The Go handler JSON is the source of truth.
- `GET /admin/primary/config` → 200 JSON: `{ config: { <16 hot fields, typed> }, bounds: { <field>_min, <field>_max for the 9 numeric fields } }`. Returned straight from the `gen.PodConfig` row; the Go handler JSON is the source of truth. 404/empty-row handling: if pod_config is unseeded (should not happen post-17-03 boot seed), return a `pod_config_unseeded` error envelope (the boot seed guarantees a row).
- `PATCH /admin/primary/config` body: `{ "field": "<hot_field_or_bound_name>", "value": <typed>, "kind": "config" | "bound" }` → 200 on success, 400 `validation_error` if out of current bound / bad format, 401 if no/bad X-Admin-Key. One field per request (clean audit diff).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dashboard server action -> gateway admin API | Crosses with X-Admin-Key (server-only). The PATCH endpoint is the only mutation path into pod_config; the GET endpoints are read-only. |
| operator-supplied value -> pod_config UPDATE | Untrusted value crosses here; validated server-side vs the live bound + field allowlist. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-11 | Tampering / DoS | unauthenticated config read/write or restart | mitigate | All three endpoints mount inside adminRouter so admin.Middleware (X-Admin-Key) gates them (Task 3); NO restart endpoint exists at all (D-02) |
| T-17-12 | Denial of Service | malicious config bricks prod (cap=0, empty days, bad value) | mitigate | Server-side validation vs the LIVE bound + cross-field rules (min<max, DownHour!=UpHour) before UPDATE (Task 2) |
| T-17-13 | Injection | dynamic column name in UPDATE | mitigate | Static field->query allowlist; unknown field -> 400; no string-built SQL (Task 2) |
| T-17-14 | Elevation of Privilege | structural edit smuggled through the write endpoint | mitigate | Allowlist contains ONLY the 16 hot fields + 9 bounds; structural columns are not in pod_config and not in the allowlist (Task 2 grep gate) |
| T-17-23 | Information Disclosure | secrets leak through the config-read endpoint | mitigate | config_read serializes ONLY the pod_config row (16 policy knobs + 9 bounds); VAST/MINIO/DSN are not columns in pod_config (Plan 17-01 T-17-03), so they cannot be returned |
| T-17-SC | Tampering | npm/pip/cargo installs | accept | No package installs (RESEARCH §Package Legitimacy Audit: N/A) |
