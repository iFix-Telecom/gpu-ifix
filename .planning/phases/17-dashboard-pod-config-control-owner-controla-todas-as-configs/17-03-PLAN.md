<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env (boot) → pod_config (DB) | First-boot seed copies env into DB once; thereafter DB is authoritative, env is disaster-recovery fallback. |
| live snapshot → provisioning | Hot edits reach the next provision only; in-flight lifecycles are immune. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-07 | Tampering | seed overwriting operator edits on every boot | mitigate | SeedPodConfig ON CONFLICT DO NOTHING (Plan 17-01) — boot seed is idempotent, existing row wins (Task 3) |
| T-17-08 | Denial of Service | structural read accidentally moved to snapshot → mis-provision | mitigate | GPU name/num + weights + images stay on r.cfg; test asserts structural NOT in snapshot (Task 1 behavior) |
| T-17-09 | Denial of Service | mid-lifecycle hot edit corrupts an in-flight provision | mitigate | snapshot read ONCE at provision-start; in-flight goroutine keeps captured values (RESEARCH §central-refactor-seam) |
| T-17-10 | DoS | DB-less boot breaks the gateway | mitigate | config.Load env fail-fast intact = typed fallback; seed/loader run after pool-connect, env never removed (D-02) |
| T-17-SC | Tampering | npm/pip/cargo installs | accept | No package installs (RESEARCH §Package Legitimacy Audit: N/A) |
</threat_model>
