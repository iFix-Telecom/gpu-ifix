# Phase 11 — UI Design Contract

> Visual and interaction contract for the Phase 11 additions to the standalone `dashboard/` Next.js 15 app (Better Auth SSO hardening — PRD-06). Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope (frontend surface):** Phase 11 is mostly backend hardening (load test, chaos, runbooks, LGPD docs). The ONLY UI surface is **PRD-06 dashboard SSO hardening** — extends the existing standalone Better Auth dashboard at `dashboard/`. CLI surfaces (`gatewayctl debug emit-error`, `gatewayctl key list`) and the `load-replay.py` terminal output are explicitly out of scope (non-visual).
>
> **Visual baseline LOCKED.** Phase 07 already shipped the dashboard look-and-feel (`07-UI-SPEC.md` `status: draft`, implemented + verified live Phase 10). Phase 11 does NOT redesign. It ADDS five auth-flow screens/states consistent with the existing login pattern. All design tokens (spacing, typography, color, components) are inherited verbatim from Phase 07 unless explicitly overridden below.
>
> **Phase 11 net-new screens / states:**
> 1. TOTP 2FA enroll flow (post-login, first-time) — QR code + manual secret + verify code + backup codes display (D-12)
> 2. TOTP 2FA challenge on login — second step after password (D-12)
> 3. Login screen — rate-limit feedback message + countdown when 5/15min/IP exceeded (D-14)
> 4. Email allowlist rejection state — when `@ifixtelecom.com.br` domain check fails on signUp (D-13)
> 5. Session-expired / idle-timeout UX — redirect to `/login` with informational message (D-15, idle 30min vs prior 7 days)

## Visual Hierarchy

The auth flow is a **single-task focus surface** — one `Card` centered in viewport (`min-h-screen items-center justify-center p-6`), exactly mirroring the existing `dashboard/src/app/login/page.tsx`. The card is the dominant element; everything else is empty dark background. This pattern is the LOCKED baseline.

**Per-screen primary focal point:**

| Screen | Focal point |
|--------|-------------|
| Login | E-mail input → password input → **Entrar** button (existing) |
| TOTP enroll step 1 (QR) | QR code image (256×256 px) + manual-entry secret string below |
| TOTP enroll step 2 (verify) | 6-digit OTP input (`input-otp` block, large monospace digits) |
| TOTP enroll step 3 (backup codes) | List of 10 backup codes (monospace, copy-to-clipboard button per code) inside `Dialog` |
| TOTP challenge on login | 6-digit OTP input (same `input-otp` block, reused for visual continuity) |
| Rate-limited login | `Alert` variant `destructive` above the form with countdown copy |
| Email allowlist rejection | `Alert` variant `destructive` above the (disabled) form with explainer copy |
| Session-expired | `Alert` variant `default` (neutral, not destructive — expiry is benign) above the login form |

The single accent-colored primary CTA per screen is the only deliberate accent draw, matching Phase 07 rule.

## Layout Constraints

**Inherited from Phase 07** (data-table row 36px, critical banner 44px min). Phase 11 adds the following component-height constraints for the new auth surfaces — these are NOT spacing tokens:

| Component | Fixed dimension | Type | Justification |
|-----------|----------------|------|---------------|
| TOTP enroll QR code | 256×256 px (16:1 ratio of qrcode lib default) | Component dimension — not a spacing token | Standard QR scanning size — large enough to scan reliably with Authenticator/1Password camera from a laptop screen; smaller (128px) causes scan failures on lower-end cameras. |
| OTP digit input box | 44×52 px per slot | Component dimension — not a spacing token | `input-otp` block default; 44px width gives a 44px touch target (WCAG AA); 52px height (44 + 8px sm vertical padding × 2 minus optical balance) accommodates the 28px Display digit. |
| Auth Card max-width | 384px (Tailwind `max-w-sm`) | Component dimension — already in `login/page.tsx` | Inherited from existing login. Single column form on desktop (no wide layout) keeps eye scan minimal. |

## Typography

**Inherited verbatim from Phase 07** (4 sizes × 2 weights, `--font-sans` Geist Sans). Per-screen mapping:

| Element | Role | Size | Weight | Line Height |
|---------|------|------|--------|-------------|
| Card title ("Gateway ifix-ai", "Configurar 2FA", "Confirmar código", "Códigos de backup") | Heading | 20px | 600 | 1.2 |
| Card description (sub-explainer text under title) | Body | 14px | 400 | 1.5 |
| Field labels ("E-mail", "Senha", "Código de 6 dígitos") | Label | 12px | 600 | 1.4 |
| Form input default text | Body | 14px | 400 | 1.5 |
| **OTP digit text** (each of 6 boxes) | Display | 28px | 600 | 1.2 — large monospace digits for at-a-glance verification |
| Backup-code text (each of 10 rows) | Label | 12px | 600 | 1.4 — `font-mono` family override (NOT `--font-sans`) for legibility of 8-char alphanumeric codes |
| Manual-entry TOTP secret (under QR) | Label | 12px | 600 | 1.4 — `font-mono` family override |
| Countdown text in rate-limit alert ("Tente novamente em {n}s") | Body | 14px | 400 | 1.5 |
| Button label ("Entrar", "Confirmar código", "Salvar e continuar", "Copiar tudo") | Body | 14px | 400 | 1.5 (button block default) |

**Tabular numerals** (`font-variant-numeric: tabular-nums`) required on: OTP digit input slots (so digits don't jitter as user types), countdown timer text, backup-code rows.

**Font family override:** OTP digits + backup codes + manual TOTP secret use the **monospace stack** (`font-mono` in Tailwind → `ui-monospace, SFMono-Regular, Menlo, Monaco, Consolas, "Liberation Mono", "Courier New", monospace`). This is NOT a new font — it's the system monospace stack used contextually for code-shaped strings. No new `next/font` declaration needed.

## Color

**Inherited verbatim from Phase 07** (`radix-nova .dark` OKLCH tokens). No new tokens this phase.

| Role | Value | Usage in Phase 11 auth screens |
|------|-------|-------------------------------|
| Dominant (60%) | `oklch(0.13 0.028 261.692)` — `--background` | Auth page background (the empty area around the centered card) |
| Secondary (30%) | `oklch(0.21 0.034 264.665)` — `--card` | Auth `Card` surface, `Dialog` backup-codes container, `Alert` background |
| Accent (10%) | `oklch(0.648 0.2 131.684)` — `--primary` (green) | See reserved-for list below |
| Destructive | `oklch(0.704 0.191 22.216)` — `--destructive` (red) | Rate-limit alert (D-14), email allowlist rejection alert (D-13), invalid-OTP error message under input |

Accent reserved for (Phase 11 auth scope):

- The single primary action button per auth step — **Entrar**, **Confirmar código**, **Salvar e continuar**, **Copiar tudo**
- Focus rings on the OTP digit input boxes + form inputs (radix-nova `--ring`)
- The "code accepted" check icon on successful OTP verify (lucide `CircleCheck`, brief sonner toast)

Accent is **NOT** used for: the "Copiar" icon button per backup-code row (uses `variant="ghost"` neutral), the manual-entry secret string (plain `--foreground`), or the QR code (neutral white-on-black render — qrcode library default).

The **Phase 07 semantic status palette** (`--status-warning` amber) is NOT used in Phase 11 auth screens — auth flows are pass/fail binary (green success or red failure); there is no "degraded" auth state.

## Copywriting Contract

**Audience:** ~4 internal Ifix operators (D-13 email allowlist `@ifixtelecom.com.br`). Language: pt-BR, operational and direct, no marketing tone. Tone matches the existing `login/page.tsx` ("Painel de observabilidade — acesso restrito à equipe de operações.").

### Primary CTAs

| Screen | CTA copy |
|--------|----------|
| Login (existing, no change) | **Entrar** |
| TOTP enroll step 1 (QR shown) | **Já escaneei, continuar** |
| TOTP enroll step 2 (verify code) | **Confirmar código** |
| TOTP enroll step 3 (backup codes shown) | **Salvar e continuar** (operator confirms they archived codes) |
| TOTP challenge on login | **Confirmar código** |
| Backup-codes dialog secondary action | **Copiar tudo** (copies 10 codes joined by `\n` to clipboard) |
| Rate-limited login (button disabled) | **Entrar** (label unchanged; button gets `disabled` attr) |

### Screen titles + descriptions

| Screen | Card title | Card description |
|--------|------------|------------------|
| Login (existing, no change) | Gateway ifix-ai | Painel de observabilidade — acesso restrito à equipe de operações. |
| TOTP enroll step 1 | Configurar 2FA | Escaneie o QR code abaixo com seu app autenticador (Google Authenticator, 1Password, Authy). |
| TOTP enroll step 2 | Confirmar código | Digite o código de 6 dígitos exibido no seu app autenticador para concluir o cadastro. |
| TOTP enroll step 3 | Códigos de backup | Guarde estes códigos em local seguro. Cada código funciona uma única vez para entrar caso você perca acesso ao app autenticador. |
| TOTP challenge on login | Verificação em duas etapas | Digite o código de 6 dígitos do seu app autenticador. |

### Error states

| Trigger | Copy |
|---------|------|
| Invalid e-mail/senha (existing, no change) | E-mail ou senha inválidos. Verifique as credenciais e tente novamente. |
| Invalid OTP code (enroll or challenge) | Código incorreto. Confirme o código atual no seu app autenticador e tente novamente. |
| Expired OTP code (>30s window) | Código expirado. Aguarde o próximo código no app autenticador. |
| Rate-limited login (D-14, 5 attempts / 15 min / IP exceeded) | Muitas tentativas. Aguarde {n}s antes de tentar novamente. — **inside `Alert variant="destructive"` above the form**; countdown updates per-second client-side; form button stays `disabled` until counter hits 0 |
| Email allowlist rejection (D-13, signUp with non-`@ifixtelecom.com.br` domain) | Cadastro restrito a contas `@ifixtelecom.com.br`. Solicite acesso à equipe de operações. — **inside `Alert variant="destructive"` on the signUp page**; form fields stay enabled but submission stays blocked server-side (defense in depth) |
| Session expired / idle timeout (D-15, 30 min idle) | Sessão encerrada por inatividade. Faça login novamente. — **inside `Alert variant="default"` above login form on redirect**; not destructive (expiry is benign, not user error) |
| Network/server error during 2FA verify | Não foi possível confirmar o código agora. Tente novamente em alguns segundos. |
| Backup codes copy success (sonner toast) | 10 códigos copiados para a área de transferência. |

### Empty / loading states

| Context | Copy |
|---------|------|
| QR code loading (first 200ms before render) | Gerando QR code… |
| Verify-button pending state | Verificando… |
| Login-button pending state (existing, no change) | Entrando… |
| Enroll step 3 — operator hasn't copied yet (informational hint, neutral muted color) | Os códigos só são exibidos uma vez. Copie e salve antes de continuar. |

### Destructive confirmations

| Action | Confirmation pattern |
|--------|---------------------|
| Sign-out (existing dashboard sidebar action) | No confirmation modal — single-click sign-out is reversible (re-login restores). Existing Phase 07 pattern unchanged. |
| Regenerate backup codes (FUTURE — NOT in Phase 11 scope) | Deferred. Phase 11 ships one-time codes display only; regeneration is v1.1. |
| Disable 2FA (FUTURE — NOT in Phase 11 scope) | Deferred. Operator with lost device uses backup code; full 2FA-disable flow is v1.1. |

**Phase 11 ships zero destructive actions on the dashboard surface.** Backend chaos tests (PRD-02 Vast DELETE, PRD-03 iptables DROP) are operator-CLI / SSH operations — not UI surfaces — and live in `RUNBOOK-INCIDENTS.md` (D-11).
