<pitfalls>
## Plan-Specific Pitfalls

**Pitfall A — Tier-2 OpenAI fallthrough is NOT proven by this plan [reviews MEDIUM #2]:**
This chaos test drops OpenRouter (tier-1) ONLY while tier-0 (Vast primary 2×RTX 3090) remains healthy. A normal-tenant request during the drop window will be served by tier-0 returning 200 — that proves the breaker isolated tier-1 correctly but does NOT exercise tier-2 (OpenAI direct). Proving tier-2 fallthrough would require BOTH tier-0 unavailable (e.g. `gatewayctl primary force-down`) AND tier-1 dropped — that is explicitly out of scope for this plan. Do NOT add an acceptance criterion of the form "normal tenant routes to tier-2" or "X-Upstream=openai-*". The honest acceptance criterion for normal tenants is "200 served by tier-0 (X-Upstream=local-llm) while tier-1 is dropped".

**Pitfall B — Do not mix natural-observation with forced-open in the same evidence pass [reviews MEDIUM #3]:**
Segment A (natural observation) proves "iptables DROP causes breaker to open by natural timeout, which causes sensitive 503". Segment B (forced-open smoke) proves "11-04 D-09 race fix works: smoke-sensitive-failover.py with `--induce gatewayctl` correctly polls OPEN_LIKE_STATES". These are separate proofs. If you invoke `smoke-sensitive-failover.py --induce gatewayctl` while iptables DROP is also active, you cannot tell from evidence whether the breaker opened from the iptables timeout OR from the gatewayctl force-open call — the smoke script's success criterion conflates the two paths. Run them sequentially with a full cleanup between.

**Pitfall C — CF IP rotation mid-test [reviews LOW #4]:**
OpenRouter resolves behind Cloudflare; the IP set returned by `dig +short openrouter.ai` can change within a 60–120 second window. If you apply the DROP rule once and walk away, the next CF rotation may route gateway traffic through an unblocked IP and the breaker will never open (or will re-close after a brief blip). Script MUST re-resolve every 30s during the chaos window and append any new IPs to the DROP rule, logging each rotation event to evidence.

**Pitfall D — Host sha256 snapshot is the canonical host-integrity proof, NOT residual grep:**
`iptables -S OUTPUT | grep -c phase11-chaos-openrouter` returns 0 even if the script wrote OTHER rules (without the tag) to the host chain. sha256 of the full `iptables -S OUTPUT -v -n` output catches ANY mutation. Use sha256 equality as the host-integrity gate; use tag-grep only as a secondary defensive check inside the container netns.
</pitfalls>
