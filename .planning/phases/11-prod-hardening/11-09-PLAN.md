    Pattern G header (5 elements):
    1. # 2FA Admin Recovery Runbook
    2. **Phase 11 (`ai-dashboard` SSO).** Read this when: an Ifix admin has lost their authenticator device AND lost their 10 backup codes; they cannot log in via TOTP challenge.
    3. **Last updated: 2026-05-27.**
    4. Related runbooks: RUNBOOK-INCIDENTS.md, RUNBOOK-DEPLOY.md (per-env keys + seed-admins.sh).
    5. ## Mental Model (30 seconds)
       2FA is enforced by Better Auth twoFactor plugin (D-12). A locked-out admin has:
       - lost their authenticator app (phone wipe, app deletion, …) AND
       - lost their 10 backup codes (no password manager copy).
       The recovery resets twoFactorEnabled to false AND deletes the twoFactor row for that user. On next login the middleware redirects to /2fa/enroll, allowing fresh enrolment.

    ## Authorization rules (separation of duty)
    - The locked-out admin requests recovery via a SECONDARY channel: WhatsApp voice call to a second admin (NOT a text-only request — confirm voice identity).
    - The recovery is EXECUTED by a DIFFERENT admin (NOT the locked-out one). Self-service recovery is forbidden — it would defeat the 2FA control.
    - Every recovery writes an audit row before the SQL UPDATE.

    ## Procedure

    ### Option A: gatewayctl helper (preferred when subcommand exists)
    If a future Phase ships `gatewayctl admin-key reset-2fa --email=<>` (TBD), prefer it — the subcommand wraps audit log + SQL update atomically. Until then use Option B.

    ### Option B: direct Postgres update (current default)
    Operator (the EXECUTING admin, not the locked-out one) runs:
    1. Get a fresh psql session against the prod dashboard DSN:
       ```
       psql "$PROD_DSN_DASHBOARD" -c "SELECT id, email, two_factor_enabled FROM dashboard_auth.user WHERE email = '{LOCKED_OUT_EMAIL}';"
       ```
       Confirm the row exists and `two_factor_enabled = true`.
    2. Write the audit row FIRST (before the reset):
       ```
       psql "$PROD_DSN_DASHBOARD" <<'EOSQL'
       INSERT INTO dashboard_auth.audit_log (actor_email, action, target_email, reason, ts)
       VALUES ('{EXECUTING_ADMIN_EMAIL}', '2fa-reset', '{LOCKED_OUT_EMAIL}', 'device-loss-and-backup-codes-lost (RUNBOOK-2FA-RECOVERY)', NOW());
       EOSQL
       ```
       (If `dashboard_auth.audit_log` does not exist on the installed schema, fall back to a comment-tagged INSERT in any audit-capable table, OR document the manual audit entry in a postmortem ticket — never skip the audit.)
    3. Run the reset in a single transaction:
       ```
       psql "$PROD_DSN_DASHBOARD" <<'EOSQL'
       BEGIN;
         UPDATE dashboard_auth.user SET two_factor_enabled = false WHERE email = '{LOCKED_OUT_EMAIL}';
         DELETE FROM dashboard_auth.twoFactor WHERE "userId" = (SELECT id FROM dashboard_auth.user WHERE email = '{LOCKED_OUT_EMAIL}');
       COMMIT;
       EOSQL
       ```
    4. Verify:
       ```
       psql "$PROD_DSN_DASHBOARD" -c "SELECT email, two_factor_enabled FROM dashboard_auth.user WHERE email = '{LOCKED_OUT_EMAIL}';"
       psql "$PROD_DSN_DASHBOARD" -c "SELECT count(*) FROM dashboard_auth.twoFactor WHERE \"userId\" = (SELECT id FROM dashboard_auth.user WHERE email = '{LOCKED_OUT_EMAIL}');"
       ```
       Expect: two_factor_enabled=false; twoFactor row count=0.
    5. Notify the locked-out admin via the secondary channel: "2FA reset committed; log in via /login, you will be redirected to /2fa/enroll to set up a fresh authenticator. Save backup codes in your password manager this time."
    6. Locked-out admin completes fresh enrolment per dashboard UI flow (11-02 enroll page).

    ## Audit trail
    The audit_log row written in step 2 is the canonical proof. If an incident review questions the reset, search:
    ```
    psql "$PROD_DSN_DASHBOARD" -c "SELECT * FROM dashboard_auth.audit_log WHERE action = '2fa-reset' ORDER BY ts DESC LIMIT 20;"
    ```

    ## Pitfalls
    - DO NOT skip the audit row — it's the only repudiation defense if recovery is later questioned.
    - DO NOT execute against your own user (separation of duty).
    - DO NOT communicate the reset over an unauthenticated text channel — voice call or password-manager-shared secure note only.
    - DO NOT reset multiple users in bulk — each reset is a single-user event with its own audit row.
