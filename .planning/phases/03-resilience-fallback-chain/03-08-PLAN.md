    ```markdown
    # Failover & Circuit Breaker Runbook

    **Scope:** Phase 3 `ifix-ai-gateway` resilience layer. Read this when:
    - `/v1/health/upstreams` shows `status != "ok"`
    - Alert fires on `gateway_breaker_trips_total` > 0 in the last 5 min
    - A client app reports 503 with `code: "upstream_unavailable*"`
    - Post-incident review of a failover event

    ## Mental Model (30s)

    The gateway has 6 upstreams, in pairs:
    - **LLM:** local-llm (tier-0, Vast.ai pod) → openrouter-chat (tier-1, Fireworks)
    - **STT:** local-stt (tier-0, pod) → openai-whisper (tier-1)
    - **Embed:** local-embed (tier-0, pod) → openai-embed (tier-1)

    Each upstream has a circuit breaker. 3 consecutive failures → OPEN; 30s cooldown → HALF_OPEN; 1 success → CLOSED.

    Policy differences:
    - **Normal tenants:** tier-0 OPEN → traffic routes to tier-1 automatically.
    - **Sensitive tenants** (data_class=sensitive): tier-0 OPEN → 3× retry waiting for tier-0 to close (4s budget); then 503 with `Retry-After: 30`. Never routed to external.
    - **Tool-call streams:** if SSE emitted `tool_calls` + upstream disconnected → 503 envelope with `code: "tool_call_partial_stream"`. Gateway does NOT retry. Agent layer must replay with fresh idempotency key.

    ## Quick Diagnosis

    ```bash
    # 1. Live per-upstream state
    curl -sS https://gateway-dev/v1/health/upstreams | jq

    # 2. Recent breaker transitions (Redis)
    redis-cli -h <redis> KEYS 'gw:breaker:*'
    redis-cli -h <redis> HGETALL gw:breaker:local-llm

    # 3. Last probe results
    docker exec ifix-ai-gateway /gatewayctl upstreams list

    # 4. Recent audit log
    psql "$PG_DSN" -c "SELECT created_at, upstream, status_code, error_code
                       FROM ai_gateway.audit_log
                       WHERE created_at > now() - interval '10 minutes'
                       ORDER BY created_at DESC LIMIT 20;"

    # 5. Prometheus metrics
    curl -sS https://gateway-dev/metrics | grep -E 'breaker|probe|sensitive'
    ```

    ## Incident Response by Symptom

    ### Symptom 1: `local-llm.state == "open"` for >2 minutes

    **Likely cause:** Vast.ai pod is dead or unreachable.

    **Diagnose:**
    1. `ssh` or web-shell into pod host.
    2. `docker ps` — llama-server container running?
    3. `curl http://<pod-ip>:8000/health` — responds?
    4. If container stopped: check `docker logs llama-server | tail -50` for OOM / crash.

    **Mitigate:**
    - If pod is stopped and recoverable: restart via `docker compose up -d llama-server` or re-run `onstart.sh`.
    - If pod is lost: Phase 6 auto-provisioning would spin a replacement. Until Phase 6 ships, manually destroy + recreate the Vast.ai instance (runbook in `.planning/phases/01-gpu-pod-image-smoke-test/01-09-SUMMARY.md`).
    - Meanwhile: normal tenants are on OpenRouter automatically; sensitive tenants are receiving 503s with `Retry-After: 30`. Client apps should back off.

    **Recovery:** Once pod healthy + `/tokenize` + `/health` pass, probe loop will close breaker within 30s (cooldown + 1 probe cycle).

    ### Symptom 2: `openrouter-chat.state == "open"`

    **Likely cause:** OpenRouter API down, Fireworks unavailable, or our API key revoked.

    **Diagnose:**
    1. Test directly: `curl -sS -H "Authorization: Bearer $OPENROUTER_KEY" https://openrouter.ai/api/v1/models | jq '.data[0]'`
    2. Check OpenRouter status page: https://status.openrouter.ai
    3. Check Fireworks status page.

    **Mitigate (critical when `local-llm` also OPEN):**
    - Normal tenants will receive 503 `upstream_unavailable` → client apps MUST back off.
    - **There is no tier-2 for chat** (D-C4 — drift Qwen→GPT is too big). Document in comms: chat is unavailable until one of (local-llm, openrouter-chat) recovers.
    - If OpenRouter itself is down: contact OpenRouter, wait.
    - If our key is the problem: rotate in Portainer stack env `UPSTREAM_LLM_OPENROUTER_AUTH_BEARER` + restart gateway.

    ### Symptom 3: Sensitive tenant reports 503s during normal load

    **Likely cause:** tier-0 breaker OPEN and sensitive retry exhausted.

    **Verify:**
    ```bash
    psql "$PG_DSN" -c "SELECT count(*) FROM ai_gateway.audit_log
                       WHERE upstream = 'blocked_sensitive'
                         AND created_at > now() - interval '5 minutes';"
    ```
    Any non-zero count means sensitive requests hit the block path.

    **Mitigate:**
    - This is **expected behavior per LGPD policy (RES-08)** — sensitive data MUST NOT go external.
    - Recover tier-0 (Symptom 1 runbook).
    - After recovery, retry-after=30 means client apps can replay safely.

    ### Symptom 4: Tool-call request returned 502 with `tool_call_partial_stream`

    **Expected behavior, not an incident.** Per RES-06 / SC-4, the gateway intentionally does not retry tool-calls. The agent layer (application code consuming the gateway) MUST:
    1. Detect the 502 envelope.
    2. Generate a NEW idempotency key.
    3. Retry the request from scratch.

    If an agent is looping on the same idempotency key, that's a client bug.

    ### Symptom 5: `gatewayctl upstreams update` didn't reload the gateway

    **Diagnose:**
    1. Did the UPDATE actually change a config column? Probe writebacks (last_probe_*) intentionally don't trigger reload (Pitfall 7 — migration 0009 WHEN clause).
    2. Is the gateway LISTEN connection alive? `psql -c "SELECT pid, query FROM pg_stat_activity WHERE query LIKE '%LISTEN%';"` — should see 1 row with query `LISTEN upstreams_changed`.
    3. Check logs for `module=LISTEN` errors.

    **Mitigate:** Restart the gateway container — listener reconnects + reloads on boot.
