### Cascading finding (NEW, not in carry-forward)

**Allowlist scope mismatch.** Operator-curated allowlist `[43803, 55158]` contains zero `1xRTX 3090` offers (`iter1-gateway-logs.log:5`). The gateway's "broaden to full qualified search" fallback at `reconciler.go` landed iter-1 on out-of-allowlist host `51096`, which happened to be the host with the port-binding defect. The cascading chain is:

```
env-precedence drift (NUM_GPUS resolved to 1 instead of 2 — see RESEARCH-ENV-PRECEDENCE)
  → allowlist queried for 1xRTX 3090 → zero matches
  → "broaden to full qualified search" fallback triggers
  → low-quality host 51096 picked
  → host never binds public ports
  → reconciler silent-waits 17 min until operator force-down
```

This means **a fix to env-precedence alone could eliminate the cascade**, but does NOT address the underlying defect (a host with this behavior would still be reachable via broaden-fallback in other scenarios). Fix Options below address each layer of the chain.

## 6. Recommended Option

**Option A (offer-side RFC1918 filter) + Option B (reconciler health-gate) bundled** — implemented together in Phase 6.6.Y.

### Rationale (evidence-tied)

- **A targets the failure class directly.** The confirmed iter-1 evidence (`iter1-vast-timeline.log:324, 327`) shows the host advertised `public_ipaddr=192.168.1.8` BEFORE the pod was even created. A picker-stage filter on RFC1918 `public_ipaddr` would have rejected `host_id=348505` from the qualified-offer set at `iter1-gateway-logs.log:6` (`offer_count=20`), forcing the next-best offer. Cheapest and most surgical fix tied directly to the four-family-confirmed root cause.
- **B is the defence-in-depth.** Even with A in place, a host could pass the filter (e.g., advertise a public IP at offer time but fail to bind ports at runtime). The current `waitForReadyOrDestroy` silent-wait (`iter1-gateway-logs.log:8-64`, 17 min of MODELS-only heartbeats) is operationally invisible. B converts that silence into a 120s-bounded fail-fast with a distinct `closure_reason`. Operator-visible, alertable, and bounded cost.

### Why NOT C alone

Option C (pre-bake / 3-way download whisper) addresses **suspect #2 (REMAINING HYPOTHESIS)** — a real concern carried over from Phase 11.2 cenários 1+5 PARTIAL, and worth doing on its own merits — but **does not address the iter-1 CONFIRMED root cause**. Per evidence, the pod never reached the download stage (`gpu_util=0`, `disk_util=23/50 GB` image-only at T+17min — `iter1-vast-timeline.log:334-335`). Without A+B, even with weights pre-baked, the next cold-start could still hit a 192.168/16-advertising host and silent-wait. C should be scheduled in Phase 6.6.Y as a parallel work item, but it is NOT the cold-start fix. Order C **after** A+B so the next live UAT actually reaches the supervisord/weights stage to validate.

### Why NOT D alone

Option D (allowlist hard-fail) depends on `06.6.X-RESEARCH-ENV-PRECEDENCE.md` landing first (shape must resolve correctly to `2×3090` for the existing allowlist `[43803, 55158]` to match anything). It also leaves a brittle operator-burden tail. Pair D with env-precedence fix in 6.6.Y but treat A+B as the cold-start primary.

### Why NOT E

Pure speculation against the current evidence. Defer indefinitely.
