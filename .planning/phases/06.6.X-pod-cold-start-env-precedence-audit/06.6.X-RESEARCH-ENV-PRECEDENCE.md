### 3.1 Why the alias is silently inert at runtime

The guard `if os.Getenv("PRIMARY_VAST_NUM_GPUS_FALLBACK") == ""` returns FALSE in production because the **gateway docker-compose.yml hardcodes a default** for the new-slot var (`06.6.X-AUDIT-RUNTIME-ENV.md:120`):

```yaml
PRIMARY_VAST_NUM_GPUS_FALLBACK: ${PRIMARY_VAST_NUM_GPUS_FALLBACK:-2}
```

So even when the operator does NOT set `PRIMARY_VAST_NUM_GPUS_FALLBACK` in `stack.env`, the compose default `2` is injected → `os.Getenv("PRIMARY_VAST_NUM_GPUS_FALLBACK")` returns `"2"` (non-empty) → the alias copy is skipped → the `slog.Warn` never fires → the operator never sees the deprecation message.

**Triple consequence:**
1. Legacy `PRIMARY_NUM_GPUS=2` operator setting is **completely dead code** at runtime (verified by `iter1-gateway-logs.log:8-64` — zero `slog.Warn` lines about deprecation in 18 minutes of logs).
2. The shape-0 (Primary) slot is governed exclusively by `PRIMARY_VAST_NUM_GPUS_PRIMARY` (compose default `1`).
3. Operator who reads MEMORY/runbook and sets only the legacy var believes they have requested 2×3090 — actually they have requested 1×3090 first, with a 2×3090 fallback that the reconciler may never reach (cold-start on a broken host blocks the iteration to shape 1).

### 3.2 Was the alias mapping target ever correct?

Per the comment block at `config.go:616-618`: *"PRIMARY_NUM_GPUS → PrimaryVastNumGPUsFallback (legacy variable historically meant 'GPUs per primary pod', which is now the FALLBACK shape per primary-gpu-shape-06.8-final memory.)"*

This mapping is **semantically defensible** because the standing 2×3090 config (per MEMORY `primary-gpu-shape-06.8-final`) is now the fallback shape — anyone who wrote `PRIMARY_NUM_GPUS=2` in pre-Phase-11.1 days was asking for "the 2×3090 pod", which is what the Fallback slot now represents.

It is **operationally broken** because:
- The shape-pair architecture iterates Primary FIRST, so the fallback shape (where the legacy value lands) is never reached during normal cold-start.
- The "copy-on-empty" guard never fires because compose defaults are non-empty.
- No documentation source teaches operators about the shape-pair model — `RUNBOOK-PRIMARY-POD.md` is silent on shape 0 vs 1 (`06.6.X-AUDIT-DOCS-VALUES.md:87-89`).

## 5. Deprecation Plan: **HARD FAIL-FAST**

Recommendation (D-13: pick ONE):

### **Recommendation: HARD FAIL-FAST at boot if ANY legacy alias is set in `os.Environ`.**

`config.Load()` should return a non-nil error with a structured message when `os.Getenv("PRIMARY_NUM_GPUS") != ""` OR `os.Getenv("PRIMARY_GPU_NAME") != ""` OR `os.Getenv("PRIMARY_VAST_PRICE_CAP_DPH") != ""`. | Operator runbook break risk | **HIGH on first deploy after fix** — every existing operator-set `PRIMARY_NUM_GPUS=2` (vps-ifix-vm, plus any prod hosts that copy `.env.prod.example`) will crash-loop the gateway until the operator migrates stack.env. | Silent drift risk | **ZERO** — gateway will not start until env is clean. | Detection latency | Immediate (gateway boot). | Indefinite (today's incident: ~30 days between Phase 11.2 close 2026-05-08 and detection 2026-06-09). |
| Reversibility | High — `git revert` restores soft warn. | n/a (no fix to revert). |
| Operator effort to remediate one host | ~2 minutes (edit stack.env, redeploy). | n/a (no remediation requested). |
| Cost of NOT fixing | Repeat of 11.2 Item 3 incident every time someone touches env. Cascades into cold-start failure (§2.4 row 1 → Plan 05 root cause). | n/a. |

### Why soft warn is rejected

The soft-warn strategy (current code) has been in production since Phase 11.1 D-A6. It has produced **zero observed deprecation log lines** (`iter1-gateway-logs.log` 18min capture) because the guard never fires (§3.1). MEMORY `primary-gpu-shape-06.8-final` (`06.6.X-AUDIT-DOCS-VALUES.md:24-27`) and `.env.prod.example:125` (`06.6.X-AUDIT-DOCS-VALUES.md:24`) both continue to teach operators the legacy var — i.e. the warn-on-set strategy has had ~30 days to migrate the ecosystem and failed.

A failing boot creates an actionable, time-bounded incident; a silent drift creates a $1.20 + 18-minute spike (this phase) plus a $0.04 cold-start failure with 1×3090 instead of 2×3090 — and an unknown number of past incidents that were attributed to "Vast flakiness".

### Mitigation for the break risk

1. Land the hard-fail change in Phase 6.6.Y with an explicit operator-prep step: SSH each host, remove legacy lines from stack.env, confirm the 6 canonical vars are present (or rely on compose defaults).
2. Update `.env.prod.example` in the SAME Phase 6.6.Y PR (lines 125-126 must drop the legacy var rows and add `PRIMARY_VAST_NUM_GPUS_FALLBACK=2`, `PRIMARY_VAST_GPU_NAME_PRIMARY=RTX 3090`, etc.).
3. Update MEMORY `primary-gpu-shape-06.8-final` to teach the canonical names.
4. Add the 6 canonical vars to `RUNBOOK-PRIMARY-POD.md` env table (today contains only `PRIMARY_VAST_PRICE_CAP_DPH=0.40` per `06.6.X-AUDIT-DOCS-VALUES.md:38-39`; that row must be replaced).
