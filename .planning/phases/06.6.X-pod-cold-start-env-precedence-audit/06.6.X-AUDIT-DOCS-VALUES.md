## Documentation Gaps (vars used in code but absent from operator-facing docs)

Per plan 03 `<action>`: feed plan 09 recommendation block.

1. **`PRIMARY_NUM_GPUS` is absent from RUNBOOK-PRIMARY-POD.md.** The runbook documents `PRIMARY_VAST_PRICE_CAP_DPH` (with `Required=Yes` column) but does not list `PRIMARY_NUM_GPUS` in the same env table at all. An operator following only the runbook would never set this var, triggering the 11.2 Item 3 drift (single-GPU pod vs 2×3090 standing config). **High-impact gap** — directly explains the carry-forward incident.
2. **`PRIMARY_GPU_NAME` is absent from RUNBOOK-PRIMARY-POD.md.** Same shape as gap #1 — runbook has no entry for the shape-selection var. Compose default falls back to 5090 (per memory line 30) instead of RTX 3090.
3. **`PRIMARY_VAST_MACHINE_ALLOWLIST` is absent from RUNBOOK-PRIMARY-POD.md.** Without the allowlist, Vast search returns ad-hoc hosts (CDI failures, broken nvidia-container-runtime — see `.env.prod.example:131-134` blocklist commentary). Runbook gives operators no instruction to pin it.
4. **`PRIMARY_VAST_PRICE_CAP_DPH` value in RUNBOOK-PRIMARY-POD.md (`0.40`) contradicts `.env.prod.example` and MEMORY (`0.60`).** The runbook value is the Phase 1 RTX 4090 era number; it was never updated when Phase 06.8 promoted 2×RTX 3090 to standing shape. An operator copying the runbook value would set the cap too low for the allowlisted hosts and the pod would never provision. **Confirmed code-vs-docs divergence inside a single source family** (this is in addition to docs-vs-runtime drift that plan 02 captures).
5. **`gateway/.env.example` documents zero shape vars.** Only image tags + whisper weights. New developers using `.env.example` as a template would never receive any shape configuration scaffolding.
6. **No doc source mentions the new canonical names (`PRIMARY_VAST_NUM_GPUS_PRIMARY`, `PRIMARY_VAST_NUM_GPUS_FALLBACK`, `PRIMARY_VAST_GPU_NAME`).** Code introduced the rename + legacy-alias warn pattern at `config.go:622-642` (per CONTEXT.canonical_refs), but every doc source still uses the legacy names. If the deprecation goal is to remove the legacy aliases, operators have no migration target documented anywhere.
7. **`RUNBOOK-OPS.md` does not exist** in the repository despite being referenced by plan 03 `<read_first>` (and by CONTEXT.canonical_refs line 100 narrative). If this file is the intended "operator env edit + recreate pattern" home, the gap is structural: there is no canonical ops runbook holding env-precedence instructions. Plan 09 should propose either creating it or merging that content into RUNBOOK-PRIMARY-POD.md.
