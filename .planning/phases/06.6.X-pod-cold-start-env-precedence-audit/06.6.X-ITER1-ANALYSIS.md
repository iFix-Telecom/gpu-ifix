1. **Top-1 suspect NOT yet confirmed by ≥2 signal families** (would test a hypothesis-fix-variant to isolate).

Pre-condition #1 is **not met**: suspect #4 (Vast host port-binding failure) is already CONFIRMED with **4 of 4 signal families** in agreement (Vast API + pod-ssh probe + network port probe + gateway-silent-wait). The CONTEXT.md D-13 phase deliverable is *diagnosis only* — no fix code lands this phase, so iter-2 cannot validate a fix-variant; it would only re-confirm what 4 signal families already proved.

The 3 REMAINING HYPOTHESES (suspects #1, #2, #3) all sit **behind** suspect #4: they require pod-side SSH reachability to instrument. Re-running iter-1 against the same offer pool would produce the same blocked-at-reachability outcome. To instrument suspects #1–#3 we would need a *different* repro setup — specifically a pod that *does* bind public ports — which means choosing a different machine / host. That's a **distinct repro variant**, not "iter-2 of the same hypothesis", and it has no fix decision riding on it because D-13 forbids code fixes this phase.

### What iter-2 *could* test (if approved later, OUT OF SCOPE here)

If Phase 6.6.Y planning concludes it needs pod-side instrumentation for suspects #1–#3 before locking a fix, a follow-up iter would:

- **Cost:** ~$0.60/h × ~30 min = $0.30 expected, within $1.16 remaining budget.

**This is captured as a Phase 6.6.Y planning input only. Not executed in 06.6.X.**

**Iter-2 SKIP.** Move to Plan 06 (env-precedence synthesis), Plan 07 (RESEARCH-COLD-START), Plan 08 (RESEARCH-ENV-PRECEDENCE). All four D-08 suspects have a verdict; the CONFIRMED root cause is sufficient to draft the RESEARCH-COLD-START fix options.
