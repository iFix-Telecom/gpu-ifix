<objective>
Run the D-01 disambiguation spike BEFORE the Option A filter heuristic is frozen. Provision ONE pod via the gateway `force-up` path against a different geo/host than iter-1's Brazil host 51096 (host_id 348505), capture the 4-signal evidence (reusing the 06.6.X method), and answer two questions that gate the code waves:

1. **Host-specific vs universal port-bind failure** (RESEARCH-COLD-START §8 Q1): does the 2nd host bind public ports, or also fail like 51096? n=2 sample. If universal → W2 must harden Option D (allowlist hard-fail) alongside Option A.
2. **Offer-side `public_ipaddr` FIELD AVAILABILITY** (PATTERNS Option A caveat): does the Vast `SearchOffers`/`bundles` *offer* object carry a `public_ipaddr` field at all? This is a FIELD-AVAILABILITY question ONLY. **Per the cross-AI review (finding #4), the spike does NOT decide the implementation site** — Option A is ALWAYS implemented client-side (a `RejectPrivateIPOffers` filter applied to the returned offer slice, mirroring `vastutil.FilterBelowCap`), because the Vast offers search-filter map has no clean `public_ipaddr` comparator. What the spike decides is whether that client-side reject is EFFECTIVE at offer time (offers carry public_ipaddr → it can drop RFC1918 advertisers pre-provision) or a NO-OP at offer time (offers lack the field → empty IP is kept, and Option B is the sole runtime catch).

Purpose: prevent freezing a filter heuristic on n=1 evidence and tell plan 6.6.Y-03 whether the client-side reject has anything to bite on at offer time.
Output: `6.6.Y-01-SPIKE-EVIDENCE.md` with the two answers + Option D directive (if applicable) + spend log.
</objective>
