## Summary

Reviewed the 9 source files changed between `b37891a` and `HEAD` for phase 6.6.Y (env-precedence hard fail-fast, Option A RFC1918 offer reject, Option B port-bind fail-fast, onstart 3-way parallel downloads, boot-time shape dump). Two Critical findings. First: Option B as implemented gates on URL *constructability* from the Vast `ports` map (`buildPodURLs` reads `inst.PublicIPAddr` + `inst.Ports`) — exactly the signal the 6.6.Y-01 spike DIRECTIVE forbade — so both observed failures (running + populated ports map + externally unreachable) bypass the 120s fail-fast entirely and still silently burn the full 40-min cold-start budget. ## Critical Issues

### CR-01: Option B fail-fast cannot catch the observed failure mode — it gates on the Vast ports map, not reachability

**File:** `gateway/internal/primary/reconciler.go:1054-1078` (also misleading comments at `reconciler.go:961-970`, `reconciler.go:1115-1125`)

**Issue:** The new port-bind fail-fast fires only when `buildPodURLs(inst)` returns any empty URL. But `buildPodURLs` → `podPortURL` (`gateway/internal/primary/lifecycle.go:407-419`) constructs URLs purely from `inst.PublicIPAddr` and `inst.Ports` — i.e. from the Vast API `ports` map. The 6.6.Y-01 spike evidence is explicit:

- Both observed failures (n=2: iter-1 machine 348505, iter-2 machine 317118) reached `actual_status=running` **with a populated `ports` map** (`8000/tcp→40554, 8001/tcp→40556, 8003/tcp→40507, 9400/tcp→40534`) and stayed externally unreachable (TCP probes from two vantage points **timed out**) for 40+ min.
- Spike DIRECTIVE (6.6.Y-01-SPIKE-EVIDENCE.md, "Answer Q1"): "the Vast `ports` field is NOT a reliable readiness signal — Option B (runtime fail-fast) MUST gate on **gateway-observable reachability (URLs probe)**, NOT on the Vast API `ports` map. A populated `ports` map with `actual_status=running` is exactly the lie this spike characterizes."

For those hosts, `buildPodURLs` returns 4 non-empty URLs on the first running poll, the new branch (`urls.LLM == "" || ...`) is never entered, and the loop falls through to the `HealthCheck` calls — which fail with a bare `continue` (lines 1086-1097, no logging) until the 40-min `health_timeout` deadline. Net effect: the shipped Option B catches only the **empty-ports-map** subclass (which neither observed failure exhibited), and the observed failure class still produces a silent 40-min budget burn — the precise behavior this phase set out to eliminate. The in-code comments ("gates on gateway-observable URL reachability ... NOT the unreliable Vast ports map", "The observed (timeout-on-public-IP) failure mode is caught by Option B ... the PRIMARY runtime catch") and the 6.6.Y-03-SUMMARY claim ("done — the fail-fast keys on buildPodURLs, not the Vast ports map") are factually wrong: `buildPodURLs` *is* the Vast ports map.

**Fix:** Once URLs are non-empty, add an actual reachability gate anchored to `firstRunningAt`: a cheap TCP probe (e.g. `net.DialTimeout("tcp", host:port, 5*time.Second)` against the LLM host:port) that distinguishes connect-timeout (host never NAT-published — the spike failure signature) from connect-refused/success (host reachable, services still booting — legitimate cold start, keep polling under the 40-min budget). Apply `public_port_bind_timeout` when contiguous running time with TCP-unreachable ports exceeds `PrimaryPublicPortBindBudgetSeconds`, and emit the same per-poll Warn in that branch (the `HealthCheck`-fail path is still completely silent today). Then correct the comments and SUMMARY claims.

```go
// after urls are non-empty, before HTTP health checks:
if !tcpReachable(urls.LLM) { // DialTimeout; timeout (not refused) == unreachable
    log.Warn("primary provisioning: running, ports published, host TCP-unreachable", ...)
    if time.Since(firstRunningAt) >= portBindBudget {
        vastutil.BestEffortDestroy(ctx, r.deps.Vast, r.deps.Log, instanceID)
        _ = r.closeLifecycle(context.Background(), lifecycleID, "public_port_bind_timeout", 0)
        return errors.New("primary: public port bind timeout")
    }
    continue
}
```

### CR-02: onstart `wait "$QWEN_PID" "$BGE_PID" "$WHISPER_PID"` swallows download/SHA-256 failures for 2 of 3 weights

**File:** `gateway/internal/primary/onstart.go:157` (script line `wait "$QWEN_PID" "$BGE_PID" "$WHISPER_PID"`; false doc claim at `onstart.go:61-62`)

**Issue:** In bash, `wait` with multiple IDs returns the exit status of the **last** ID only (`man bash`: "If multiple ids are supplied, the return status is the exit status of the last id waited for"). `set -e` therefore only aborts if the *whisper* download fails. If the Qwen or BGE-M3 `download_with_verify` background job fails — aria2c exhausting 50 retries, or critically a **SHA-256 mismatch** (`sha256sum -c` nonzero) — its nonzero status is discarded, the script logs "3 downloads complete", and proceeds to `exec supervisord` with a missing or corrupt/tampered `model.gguf` / `model.tar.gz`. This silently defeats the documented integrity fail-fast ("`set -e` propagates aria2c / sha256sum / tar failures so a single bad weight download aborts the pod — T-06.6-02 mitigation") for 2 of the 3 weights, and downstream the pod burns the full 40-min cold-start budget (llama-server can't load weights, health never passes) instead of failing immediately. The bug predates this phase (2-PID variant had the same flaw for Qwen) but this phase rewrote the exact line and extended it to 3 PIDs.

**Fix:** Wait on each PID individually so `set -e` aborts on the first failure:

```bash
echo "[$(date -u +%Y-%m-%dT%H:%M:%SZ)] onstart: waiting for 3 downloads"
wait "$QWEN_PID"
wait "$BGE_PID"
wait "$WHISPER_PID"
```

## Warnings

### WR-01: `isRFC1918` covers only RFC1918 — loopback, link-local, CGNAT, and 0.0.0.0 advertisers pass Option A — and reimplements stdlib

**File:** `gateway/internal/emerg/vast/types.go:68-89`

**Issue:** A Vast host advertising `public_ipaddr` of `127.0.0.1`, `169.254.x.x` (link-local), `100.64.0.0/10` (CGNAT — common behind carrier NAT, never externally reachable), or `0.0.0.0` is just as provably unreachable as an RFC1918 address, yet passes the Option A filter and costs a full provision + Option B budget to discover. Additionally, the hand-rolled octet comparison duplicates `net.IP.IsPrivate()` (stdlib since Go 1.17, exactly RFC1918 + RFC4193).

**Fix:** Use stdlib and broaden to all non-globally-routable classes:

```go
parsed := net.ParseIP(ip)
if parsed == nil { return false }
return parsed.IsPrivate() || parsed.IsLoopback() ||
    parsed.IsLinkLocalUnicast() || parsed.IsUnspecified()
// optionally: CGNAT 100.64.0.0/10 via netip.Prefix check
```

(Rename the function/log label accordingly, e.g. `isNonRoutable`.)

### WR-02: `PrimaryVastRejectPrivateIP` parses `!= "false"` — inconsistent with the `boolOr` helper used by every other bool in this file

**File:** `gateway/internal/config/config.go:491`

**Issue:** Every other boolean env in `config.go` goes through `boolOr` (`config.go:750-754`), which accepts `true/false/1/0/yes/no` case-insensitively. `PRIMARY_VAST_REJECT_PRIVATE_IP` alone requires the literal lowercase string `"false"` — an operator setting `FALSE`, `0`, or `no` (all accepted everywhere else in the same file) silently leaves the filter ENABLED with no warning. The opt-out-default-true intent is fully preserved by `boolOr(os.Getenv("PRIMARY_VAST_REJECT_PRIVATE_IP"), true)`.

**Fix:** `PrimaryVastRejectPrivateIP: boolOr(os.Getenv("PRIMARY_VAST_REJECT_PRIVATE_IP"), true),` and update the struct comment + the default/override tests.

### WR-03: No validation on `PrimaryPublicPortBindBudgetSeconds` — 0 or negative in production destroys every pod on its first running poll

**File:** `gateway/internal/config/config.go:490`, `gateway/internal/primary/reconciler.go:1070-1076`

**Issue:** The `>=` comparison (deliberately chosen so budget 0 fires immediately for the deterministic test) means a production env typo (`PRIMARY_PUBLIC_PORT_BIND_BUDGET_SECONDS=0`, or any unparseable-then-`atoiOr`-defaulted edge plus a negative value) turns the fail-fast into "destroy every pod whose ports aren't bound on the very first running poll" — pods legitimately need seconds-to-minutes for the ports map to populate. There is no `Load()`-time floor and no warning. The test-only semantics of budget 0 leak into production config space.

**Fix:** In `Load()`, reject or clamp non-positive values with a logged floor, e.g.:

```go
if cfg.PrimaryPublicPortBindBudgetSeconds < 1 {
    return Config{}, fmt.Errorf("%w: PRIMARY_PUBLIC_PORT_BIND_BUDGET_SECONDS must be >= 1", ErrMissingEnv)
}
```

and have the tests inject budget 0 directly on the struct (they already build `cfg` via `testCfg(t)` mutation, so nothing breaks).

### WR-04: `waitForReadyOrDestroyForTest` test seam duplicates the production loop and has materially drifted — older tests now assert dead-code behavior

**File:** `gateway/internal/primary/reconciler_test.go:1752-1833`

**Issue:** The pre-existing copy-paste test seam no longer mirrors production: it (a) has no Option B port-bind logic at all, (b) omits the STT URL/health checks (`urls.LLM/TTS/DCGM` only, line 1815), (c) closes immediately on a single `IsTerminal()` observation (line 1806) where production requires 3 consecutive strikes, and (d) has no `status_msg` truncation parity. Seven existing tests (lines 580-1124) exercise this fork instead of `waitForReadyOrDestroy`, so they validate behavior production no longer has. This phase introduced the superior seam (`primaryInstancePollIntervalForTest` var override) that drives the real function — making the duplicate both stale and redundant.

**Fix:** Delete `waitForReadyOrDestroyForTest` and migrate its 7 callers to `withTestPollInterval(t, …)` + the production `waitForReadyOrDestroy` (the two new 6.6.Y tests already demonstrate the pattern, including the `stampFailure` concern via the production caller path).

## Info

### IN-01: `primaryInstancePollIntervalForTest` is an unsynchronized package-level mutable read by production goroutines

**File:** `gateway/internal/primary/reconciler.go:99-106`, `gateway/internal/primary/reconciler_test.go` (`withTestPollInterval`)

**Issue:** Tests mutate the var while reconciler goroutines spawned by earlier tests (via `Start`) may still be polling — Go runs same-package tests sequentially, but goroutines can outlive their test, making this a latent `-race` flake. It also leaves a mutable knob in the production binary.

**Fix:** Make the interval a `Reconciler` field defaulted in the constructor (`r.pollInterval`), overridden per-instance in tests; or guard with `atomic.Int64` nanoseconds.

### IN-02: `unsetLegacyPrimary` doc comment contradicts its own body

**File:** `gateway/internal/config/config_test.go:16-36`

**Issue:** The comment states "t.Cleanup is not needed" but the function registers a `t.Cleanup` restore three lines later (correctly — the restore IS needed to avoid leaking unsets into sibling tests when the var pre-existed). Harmless, but the comment will mislead the next editor into removing the restore.

**Fix:** Reword: "t.Cleanup restores any pre-existing value so sibling tests are unaffected."
