## Shape-0 mismatch analysis + PROACTIVE resolution (finding #1)

**The mismatch:** shape 0 demands `num_gpus=1` but the allowlist hosts (43803/55158) are 2×3090-only
(and not even rentable as 2×3090 today). So the shape-0 allowlist pass ALWAYS returns zero → the
reconciler broadens to the full qualified search → it picks an arbitrary unvetted host → which (per
the n=2 spike) becomes `actual_status=running` with a populated `ports` map yet stays externally
unreachable. The allowlist provides no protection on the shape-0 path.

**PROACTIVE resolution chosen — SPIKE-EVIDENCE Option D (allowlist hard-fail), paired with (c).**

This is the resolution the 6.6.Y-01 SPIKE-EVIDENCE directive mandates because the failure is
**UNIVERSAL (n=2)**: the Vast `ports` map is NOT a reliable readiness signal, so unconstrained
broadening into arbitrary hosts is the dangerous behavior, not a safe fallback. "Broadening is safe"
is therefore explicitly rejected as a standalone resolution (finding #1).

Concretely, recommended to plan 6.6.Y-03's allowlist path:

1. **Option D — allowlist hard-fail (PRIMARY resolution, per SPIKE-EVIDENCE universal-failure directive):**
   When an allowlist is configured, the reconciler MUST NOT silently broaden to the full qualified
   search on allowlist-exhaustion. Instead, treat allowlist-exhaustion as a **provision failure →
   Cooldown** (no pod created), with a structured log line
   (`msg="primary allowlist exhausted; hard-fail (no broaden)" shape=<n>`). This stops the gateway
   from picking arbitrary unvetted hosts that produce the running-but-unreachable lie. Broadening
   may remain available only behind an explicit operator opt-in env
   (e.g. `PRIMARY_VAST_ALLOW_BROADEN=false` default), NOT the implicit default.

2. **Paired with (c) — skip the shape-0 allowlist pass when no 1×3090 allowlist entry exists:**
   Because the allowlist hosts are 2×3090-only, the shape-0 pass against the allowlist is
   structurally always-empty. The reconciler should recognize "allowlist has no entry capable of
   shape 0" and skip the shape-0 allowlist pass (go straight to the shape-1 FALLBACK allowlist pass)
   rather than broaden. This keeps provisioning constrained to the vetted 43803/55158 hosts for the
   shape they CAN serve (2×3090 = FALLBACK).

3. **Defense-in-depth already landed in this phase:** Option A (`PRIMARY_VAST_REJECT_PRIVATE_IP=true`,
   offer-time RFC1918 drop) + Option B (`PRIMARY_PUBLIC_PORT_BIND_BUDGET_SECONDS=120`, runtime
   gateway-reachability fail-fast with `closure_reason=public_port_bind_timeout`). Per SPIKE-EVIDENCE,
   Option B (URLs-probe gating, NOT the Vast `ports` map) is the primary catch for the
   timeout-on-public-IP failure mode; Option A only catches the RFC1918 subclass.

**Net:** Option D (no implicit broaden) + (c) (skip structurally-empty shape-0 allowlist pass) is the
proactive resolution; Option B is the runtime safety net for any host that still slips through.
