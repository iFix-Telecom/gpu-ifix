# Cross-AI Plan Review: Phase 6.6.Y

## Summary
The implementation plans for Phase 6.6.Y are exceptionally well-structured, directly addressing the root causes identified in the Phase 06.6.X research. The strategy of bundling a **hard fail-fast for legacy environment variables** with **runtime health-gates for Vast.ai host port-binding** is sound and addresses both configuration drift and infrastructure flakiness. The wave-based approach correctly prioritizes a "disambiguation spike" to freeze implementation details for the RFC1918 filter and enforces a safety-critical rollout sequence to prevent gateway crash-loops.

## Strengths
- **Empirical Gating:** Plan 01 (Wave 0) uses a low-cost spike to resolve ambiguity between host-specific and universal failures, ensuring the implementation of Option A (RFC1918 filter) is built on current Vast API reality.
- **Safety-Critical Rollout:** The plan explicitly orders the Portainer stack environment migration (Plan 05) BEFORE the fail-fast image deployment (Plan 06), mitigating the risk of a "denial of service" crash-loop on the live `dev` stack.
- **Pattern Adherence:** The technical tasks rigorously follow existing in-file patterns (e.g., sentinel errors in `config.go`, strike-counters in `reconciler.go`), ensuring architectural consistency.
- **Comprehensive Observability:** The addition of a boot-time resolved-shape log dump and per-strike logs in the reconciler significantly improves the operator's ability to debug "silent hangs."
- **Budgetary Guardrails:** Inclusion of a cumulative spend log against a hard $2.00 cap for Vast.ai expenses is an excellent operational constraint.

## Concerns
- **Allowlist/Shape Mismatch (MEDIUM):** Plan 04 identifies a potential cascade where Shape 0 (1×3090) always fails the allowlist (which contains only 2×3090 hosts), triggering a "broaden" to lower-quality hosts. While the plan notes this, if the resolution is simply to "accept broadening," we remain dependent on the effectiveness of the RFC1918 filter rather than the security of the allowlist.
- **GHCR Connectivity (MEDIUM):** The plan acknowledges that GHCR push from GHA is broken org-wide. Relying on a manual PAT-retag fallback (Plan 06) is a high-friction step that could lead to "stale image" traps if the operator misses a digest comparison.
- **Option C Complexity (LOW):** Adding a 3rd parallel download (Whisper) to the `onstart.go` shell script increases the surface area for shell-script regressions (e.g., PID tracking, wait logic). The length-regression test (<14KB) is a good guard, but shell logic is notoriously brittle.

## Suggestions
- **Allowlist Proactiveness:** In Plan 04 Task 2, strongly prefer "Option A" (populating the allowlist with 1×3090 hosts) or "Option C" (skip shape 0 if allowlist is empty) rather than just documenting that broadening is safe. Reducing "broaden" events is the best way to avoid the defective host class entirely.
- **Digest Verification Automation:** In Plan 06, provide the exact `docker manifest inspect` or `skopeo` command for the operator to verify the GHCR digest before pulling to the stack, reducing the risk of a manual retag error.
- **Logging Clarity:** Ensure the `public_port_bind_timeout` closure reason includes the `host_id` and `machine_id` in the DB row/log so that recurring "bad hosts" can be easily identified and added to the `PRIMARY_VAST_MACHINE_BLOCKLIST`.

## Risk Assessment: MEDIUM
The overall risk is **Medium**. While the code changes are surgical and well-tested, the "hard fail-fast" on environment variables makes this a breaking update for the existing deployment. Success hinges on the operator's execution of the Portainer stack migration (Plan 05) in the correct sequence. The technical complexity is low, but the operational coordination required is significant.

---
**Verdict:** Approved for execution, provided the operator strictly follows the Wave 3 (Plan 05) gate before proceeding to Wave 4.

## Codex Review

## Summary

Overall, the phase is well decomposed and the plans are unusually explicit about ordering, live-risk gates, spend control, and evidence capture. The core goals are covered: env-precedence hard fail-fast, RFC1918/port-bind cold-start mitigation, operator-safe stack migration, deploy verification, and live UAT. The main risks are not missing intent, but inconsistencies between plans, a few technically questionable implementation details, and some scope expansion that could make execution brittle. The most important fixes are to normalize the canonical shape values across plans, resolve the `public_ipaddr` “search-filter map vs client-side reject” contradiction, tighten testability of the reconciler timeout path, and make the external-memory edit / live Vast API work explicitly gated where sandbox or secrets may block execution.

## Strengths

- Clear wave ordering with good safety gates: spike before code, env migration before fail-fast deploy, deploy before UAT.
- Strong attention to operational evidence: digest checks, boot-shape logs, spend logs, lifecycle closure reasons, no-orphan checks.
- Good failure-mode targeting: the plans directly address the confirmed 06.6.X failure mode instead of broad refactoring.
- Security/threat sections are pragmatic and acknowledge the Vast `?api_key=` constraint.
- The hard fail-fast migration is correctly treated as deployment-sensitive, not just a code change.
- UAT design is realistic: it allows A/B not to trigger if no bad host appears, while relying on unit coverage.

## Concerns

- **HIGH: Canonical shape defaults conflict across plans.**  
  Plan 04 `.env.prod.example` says primary shape is `1×3090` at `$0.30` and fallback is `2×3090` at `$0.60`, while Plan 05 says set `PRIMARY_VAST_NUM_GPUS_PRIMARY=2`, `PRIMARY_VAST_PRICE_CAP_PRIMARY=0.60`, “plus fallback slots.” Plan 06/07 expect boot logs showing the `2×3090 fallback shape`. This needs one canonical truth. Otherwise the live env, docs, and UAT expectations can diverge again.

- **HIGH: Option A branch logic is internally inconsistent.**  
  Plan 01 says the spike decides “types.go search-filter map vs reconciler.go client-side reject.” Plan 03 later says even when offers carry `public_ipaddr`, the filter is applied client-side because Vast has no clean comparator. That makes the spike’s implementation-site decision partly obsolete. The plan should explicitly change the branch to: “client-side reject if offer field exists; no-op-plus-Option-B if it does not.”

- **HIGH: Plan 04 modifies a file outside the workspace writable root.**  
  `/home/pedro/.claude/projects/.../memory/primary-gpu-shape-06.8-final.md` is outside the project root. Under the stated permissions, an autonomous plan may not be able to edit it. This should be operator-gated, moved into repo-owned docs, or marked as best-effort/manual.

- **HIGH: Plan 04 live allowlist refresh is marked autonomous but requires secrets/network.**  
  It needs `VAST_AI_API_KEY` and live Vast API access. Given restricted network/secret assumptions, this is not reliably autonomous. It should be `autonomous: false` or split into an operator-gated capture plus an auto doc-write step.

- **MEDIUM: Plan 03 timeout tests may be hard to write deterministically.**  
  `waitForReadyOrDestroy` appears ticker/timer driven. If there is no fake clock or injectable poll interval, tests for a 120s budget can become slow or flaky. The plan should require using a tiny env budget, injectable clock/ticker, or existing test hooks.

- **MEDIUM: Config fail-fast only checks non-empty env vars.**  
  That matches current wording, but if Compose passes empty strings for legacy rows, the behavior depends on `os.Getenv` vs `LookupEnv`. Plan 04 removes empty defaults, but tests should define whether `PRIMARY_NUM_GPUS=` should hard-fail. For strict “legacy var present,” `LookupEnv` is more accurate.

- **MEDIUM: Plan 06 says Qwen + bge-m3 + whisper, but project context says BGE-M3 is no longer on-pod.**  
  The active architecture says BGE-M3 moved off-pod to CPU Infinity. If onstart still downloads bge-m3 for another reason, the plan should justify it. Otherwise Option C risks reintroducing stale weight work into the pod cold start.

- **MEDIUM: Plan 06 assumes Whisper env config already exists and is forwarded.**  
  It says verify forwarding and add if missing, but `files_modified` only lists `onstart.go`. If forwarding is missing, this plan will need to touch additional files. Add them to `files_modified` or split the task.

- **MEDIUM: Plan 05 intentionally contradicts D-04 by preserving `VAST_PRICE_CAP_DPH`.**  
  The correction is probably right because Plan 02 identifies it as emerg subsystem config, but this should be explicitly approved or recorded as a decision update. Otherwise executor may face conflict between “locked D-04” and Plan 05.

- **LOW: Some grep-based acceptance criteria can produce false positives/negatives.**  
  Examples: searching runbook non-comment lines for `PRIMARY_GPU_NAME` may match canonical `PRIMARY_VAST_GPU_NAME_PRIMARY`; broad regexes may fail because substrings overlap. Use anchored patterns for exact legacy names.

- **LOW: Threat models accept query-param API key exposure repeatedly.**  
  This is forced by Vast behavior, but plans should add concrete hygiene: avoid shell history where possible, avoid pasting full URLs into repo evidence, redact keys from logs.

## Suggestions

- Add a short “canonical shape matrix” at the top of the phase context and reference it from Plans 02, 04, 05, 06, and 07. Decide whether `2×3090` is primary or fallback and make all env examples, boot-log expectations, and UAT checks match.

- Rewrite Plan 03 Option A acceptance to remove the search-filter-map branch unless Vast actually supports filtering by `public_ipaddr`. A better rule: inspect offers; if `public_ipaddr` exists, apply `RejectPrivateIPOffers` client-side before cap filtering; if absent, log that offer-time rejection is unavailable and rely on Option B.

- Change Plan 04 to `autonomous: false` for the allowlist refresh, or split it:
  - 04A autonomous docs/env edits inside repo.
  - 04B operator-gated Vast inventory capture.
  - 04C auto evidence file from pasted results.

- Make external MEMORY modification manual or replace it with a repo-local memory/update note. Do not have an autonomous repo plan depend on writing outside the workspace.

- In Plan 02, use `os.LookupEnv` for legacy hard-fail if the intended policy is “presence,” including empty values. Add tests for both `PRIMARY_NUM_GPUS=2` and `PRIMARY_NUM_GPUS=`.

- In Plan 03, require deterministic timeout tests. If the reconciler lacks time injection, set `PrimaryPublicPortBindBudgetSeconds=0` or `1` and use a short poll interval hook already present; otherwise add minimal test-only injection.

- In Plan 06, verify whether BGE-M3 should still be downloaded on-pod. If not, Option C should be “Qwen + Whisper + templates/service setup” rather than “Qwen + bge-m3 + whisper.”

- Add an explicit rollback path for Plan 06 deploy: previous image digest, how to redeploy it, and what evidence requires rollback.

- Redact secrets in all evidence artifacts. Add acceptance criteria that evidence files must not contain `VAST_AI_API_KEY` values or full query URLs with keys.

- Add a final consistency check before UAT: `docker inspect` env, boot-shape log, `.env.prod.example`, and runbook all agree on shape and caps.

## Per-Plan Notes

### Plan 01

Strong spike design and good evidence requirements. Main issue is that its “types.go map vs reconciler.go” decision is later contradicted by Plan 03. It should decide field availability, not necessarily implementation site. Risk: **MEDIUM**, because the spike is live but small and bounded.

### Plan 02

Good plan for hard fail-fast and boot-shape visibility. The key improvement is using `LookupEnv` if legacy presence should fail even when empty. Also ensure tests cover emerg `VAST_PRICE_CAP_DPH` separately. Risk: **MEDIUM**, because config boot behavior is high-impact but well tested.

### Plan 03

Correctly targets the failure mode, but has the biggest implementation ambiguity. The RFC1918 filter site needs to be simplified and timeout tests need deterministic timing. Risk: **HIGH**, because this is core behavior and a bad implementation can either leak cost or reject good hosts.

### Plan 04

Useful doc/env cleanup, but not fully autonomous due to live Vast API and external MEMORY file. Also grep checks need exact legacy matching. Risk: **MEDIUM-HIGH**, mostly from execution environment and consistency issues.

### Plan 05

Good safety gate before fail-fast deploy. The preserved `VAST_PRICE_CAP_DPH` correction is sensible, but it conflicts with locked D-04 wording and should be elevated as an explicit decision. Risk: **MEDIUM**, because manual env edits are error-prone but well verified.

### Plan 06

Digest verification and clean-boot proof are strong. Main concerns are stale BGE-M3 assumptions, potentially missing env-forwarding file edits, and lack of rollback. Risk: **MEDIUM-HIGH**, because deploy + onstart changes can block UAT or cold start.

### Plan 07

Good UAT closure plan with spend and orphan controls. It properly allows A/B to be “not triggered” if no defective host appears. Add final consistency checks and secret-redaction requirements. Risk: **MEDIUM**, because live GPU UAT has inherent variability but the plan handles retries and evidence well.

## Risk Assessment

Overall risk: **MEDIUM-HIGH**.

The phase goal is achievable and the plans are directionally strong, but several inconsistencies could reintroduce the exact class of drift this phase is meant to eliminate. The highest-risk items are canonical shape disagreement, Option A implementation ambiguity, and assumptions around live/external operations being autonomous. Fixing those before execution would bring the overall risk down to **MEDIUM**.

- Operational evidence discipline: digest verification, boot-shape log dump, cumulative Vast spend log vs $2.00 cap, lifecycle closure reasons.
- Plans target the confirmed 06.6.X failure mode directly — no broad refactoring, surgical changes following existing in-file patterns.
- Realistic UAT design: A/B allowed to not trigger if no defective host appears, with unit coverage as fallback.

### Agreed Concerns
1. **Allowlist/shape mismatch cascade (both reviewers — Codex HIGH, Gemini MEDIUM):** Plan 04 shape-0 (1×3090) vs allowlist (2×3090-only) inconsistency, plus canonical shape values conflicting across Plans 04/05/06/07 (`1×3090 $0.30` vs `2×3090 $0.60` as primary). Needs one canonical shape matrix referenced by all plans; prefer proactive allowlist fix over "accept broadening".
2. **Deploy/image-path friction (both — MEDIUM):** GHCR push broken org-wide → manual PAT-retag fallback risks stale-image trap. Both ask for explicit digest-verification commands; Codex additionally asks for a rollback path (previous digest + redeploy procedure) in Plan 06.
3. **Option C / onstart.go surface area (both — Gemini LOW, Codex MEDIUM):** 3rd parallel download increases shell-script regression risk; Codex flags possible stale BGE-M3 assumption (BGE-M3 moved off-pod) and missing env-forwarding files in `files_modified`.

- **Option A implementation site:** Codex flags HIGH internal inconsistency (Plan 01 spike decides types.go-map vs client-side reject, but Plan 03 already commits to client-side) — Gemini does not raise it. Codex suggestion: spike decides field availability only, implementation is always client-side reject.
- **Plan 04 autonomy:** Codex flags HIGH that live Vast API refresh + external MEMORY-file edit cannot run autonomously under workspace/secret constraints — Gemini silent. Suggested split: repo edits autonomous, live capture operator-gated.
- **LookupEnv vs Getenv (Codex only, MEDIUM):** legacy-var hard-fail semantics for empty-string values (`PRIMARY_NUM_GPUS=`) undefined; `os.LookupEnv` if policy is "presence".
- **D-04 conflict (Codex only, MEDIUM):** Plan 05 preserving `VAST_PRICE_CAP_DPH` contradicts locked D-04 wording — needs explicit decision update.
- **Overall risk:** Gemini MEDIUM (approved with Wave 3 gate); Codex MEDIUM-HIGH (fix shape canon + Option A ambiguity + autonomy assumptions first → drops to MEDIUM).
