# Phase 06.9: OpenRouter model-rewrite per-upstream — Pattern Map

**Mapped:** 2026-05-24
**Files analyzed:** 9 (1 NEW migration, 5 MODIFY Go source, 3 NEW/EXTEND tests)
**Analogs found:** 9 / 9 (all matches in-repo)
**Read-only constraint honored:** only this PATTERNS.md was written.

## Headline finding

The fix is NOT a clean drop-in for the existing `models.Handler` wire-up. Today `gateway/cmd/gateway/main.go:1213` wraps the chat handler with `models.Handler(resolver, "llm", chatHandler)` BEFORE the dispatcher runs — so the resolver gets `upstream="llm"` (the ROLE, not the resolved upstream NAME). That value also flows into `models.RewriteJSONModel(body, resolver, "llm")` at `rewrite.go:21`. The pivot to per-upstream-name resolution (D-03 in CONTEXT) requires either:

  (a) moving the model rewrite INSIDE each tier-1 director (so the resolved upstream name is known) — same shape as `BuildOpenRouterDirector` already does for `provider.order` injection, OR

  (b) deferring the rewrite to after dispatcher selection by stripping `models.Handler` from the chat/embed wire-up and inserting a new rewrite step inside `dispatchTo`/`dispatchOverride` (after `auditctx.WithBillingUpstream` already stamps the resolved upstream name).

Plan (a) is the lower-blast-radius option and matches the seed Option B description (D-01) when read literally: dispatcher already knows the upstream → resolver gets per-upstream-name target → director rewrites. The schema migration (Decision D-02) ships independent of which dispatch path consumes it.

Critical anti-pattern to flag for the planner: the current `models.Handler` wrap site uses a hardcoded role string ("llm"/"embed"); per Decision D-03 this becomes WRONG once tier-1 upstreams exist for the same role (`qwen` on `local-llm` vs `qwen` on `openrouter-chat`). Removing or refactoring the `main.go:1211-1218` block is mandatory.

### `gateway/internal/models/resolver.go` (MODIFY)

**Analog:** itself (the entire file).

**KEY insight:** the `aliasKey` struct at lines 26-29 already has the shape we need (`{Alias, Upstream}`), and the docstring says "the same alias can legitimately exist on multiple upstreams without collision (Codex review [MEDIUM] 02-05)". **Pattern excerpt (lines 26-69 — Refresh + aliasKey shape):**
```go
type aliasKey struct {
    Alias    string
    Upstream string // Phase 06.9: now upstream NAME ("local-llm" | "openrouter-chat" | ...) not role
}

// Refresh reads the model_aliases table into a fresh map and atomically
// swaps it in under write-lock.
func (r *Resolver) Refresh(ctx context.Context) error {
    rows, err := r.q.ListModelAliases(ctx)
    if err != nil {
        return err
    }
    fresh := make(map[aliasKey]string, len(rows))
    for _, row := range rows {
        fresh[aliasKey{Alias: row.Alias, Upstream: row.UpstreamName}] = row.Target
    }
    r.mu.Lock()
    r.aliases = fresh
    r.mu.Unlock()
    r.log.Info("model aliases refreshed", "count", len(fresh))
    return nil
}
```

**Conventions:**
- Keep `Resolve(alias, upstream string) string` signature — but document that `upstream` is now the upstream NAME, not role. Update the godoc at line 91-93.
- Keep the pass-through behavior (line 98-100): unknown `(alias, upstream)` → return alias unchanged. This is the safety net for new tier-1 upstreams an operator hasn't seeded yet.
- Preserve the 60s ticker (`refreshInterval` const line 19) — no LISTEN/NOTIFY for `model_aliases` exists in the codebase (only `upstreams_changed`, `tenants_changed`, `fx_rates_changed` channels per `grep LISTEN`); 60s polling is the standing pattern.
- Concurrency: keep `sync.RWMutex` (line 40) — never lock during DB I/O (current Refresh builds `fresh` outside the lock).

**Anti-patterns:**
- Do NOT add a LISTEN/NOTIFY trigger for `model_aliases` in this phase — `upstreams.listen.go` is the precedent for that scope of change and CONTEXT D-02 (additive, no breaking change) does not include it. Defer to a future phase.
- Do NOT change the `upstream` column meaning in the existing struct field; the Go map key now reads from the NEW `upstream_name` column.

**Pattern excerpt (lines 1-36):**

```go
// The "model" field selection (whisper-1) is the caller's responsibility:
// when routed to OpenAI Whisper the client MUST send model=whisper-1 in

        // (with boundary) survives because BuildDirector does not touch it.

```

**Phase 06.9 implications — flag for planner:**
- Schema migration row `('whisper', 'stt', 'whisper-1', 'openai-whisper')` is correct but DOES NOTHING until a multipart-aware rewriter exists. Two paths:
  1. **Implement multipart form rewrite** — re-encode the multipart body, replacing the `model` part. Significantly more code than JSON rewrite; new helper in `proxy/audio.go` or similar.
  2. **Document client-side responsibility** — client MUST send `model=whisper-1` in the form when local-stt breaker is OPEN. This is the current Phase 2 behavior and is fragile for SC-2.
- D-04 in CONTEXT lists openai_whisper as a director that should get the same rewrite-from-resolved-model treatment; the planner must decide between (1) and (2) and split the deliverable accordingly. Recommend deferring (1) to a sub-plan if cost > 1h.

**Anti-patterns:**
- Do NOT add JSON body rewrite to the whisper director — multipart bodies are not JSON. The Content-Type header carries the boundary; rewriting it requires `mime/multipart.Reader`.

### Breaker force-override seam (Phase 06.9 Plan 04 Task 2 — WARNING-4 closure)

**Mapped 2026-05-24 during execution. Documents the canonical FSM read site, the patch shape, and the operator semantic so future contributors don't have to re-derive the seam.**

**Canonical FSM read site:**
`gateway/internal/breaker/breaker.go:103 — Set.Execute(name string, fn func() (*http.Response, error)) (*http.Response, error)`

The breaker package uses `sony/gobreaker/v2`. There is NO custom `IsOpen()` or `Evaluate()` method — state is read via `cb.State()` and the canonical hot-path read site is `Set.Execute`, which already short-circuits via the `remoteOpen` overlay (`breaker.go:111-113`) BEFORE invoking `cb.Execute(fn)`. Phase 06.9 Plan 04 Task 2 adds a force-override check that takes PRECEDENCE over both `remoteOpen` and the local gobreaker state — operator intent wins over peer observation.

**Breaker struct Redis surface:**
ALREADY INJECTED. `breaker.go:42 rdb *redis.Client` field + `NewSet(rdb *redis.Client, ...) *Set` constructor at line 52. NO constructor refactor was needed — we reuse the existing `rdb` field for force-override reads, matching the existing publish path that also uses `s.rdb` in `publishTransition`.

**Eval cadence:**
Per-request through `Set.Execute`. There is NO background ticker. The existing `remoteOpen` overlay pattern is a per-request lookup of an in-memory map populated by Pub/Sub (`subscribe.go` publishes into `s.remoteOpen` via `applyRemoteEvent`). The Phase 06.9 force-override mirror this shape: an in-memory cache (`Set.forceCache`) populated lazily by `Set.maybeRefreshForceOverride` with a 1-second freshness debounce.

**Cache + Redis read pattern:**
- `Set.forceCache` is a separate `*forceCache` struct with its own `sync.RWMutex` (kept independent of `Set.mu` so cache lookups never contend with `Rebuild` / `Snapshot`).
- `CheckForceOverride(name)` is a pure cache read (~50ns p99) — used on the request hot path inside `Execute`.
- `RefreshForceOverride(ctx, name)` performs a single Redis GET + JSON decode + PTTL fetch via `ReadForceOverride` and updates the cache. ~30-100µs against miniredis; ~1-3ms against real Redis (well under the WARNING-4 ≤1ms-per-tick budget when amortized across the 1s freshness window).
- `maybeRefreshForceOverride(name)` is the lazy refresh: returns immediately when `time.Since(entry.lastRefresh) < forceCacheFreshness` (1 second); otherwise calls `RefreshForceOverride`. The first Execute per upstream per second pays the Redis cost; subsequent calls within the second are cache hits.

**Operator semantic (Plan 06 UAT consumer):**
- `gatewayctl breaker force-open --upstream X --ttl 60s` (Plan 04 Task 3) writes `gw:breaker:force:X` with JSON value + Redis EX. Within ~1 second (the cache freshness window), the next request to upstream X short-circuits with `ErrBreakerOpen` — without firing the upstream request. Operator can run multi-step UAT scenarios with deterministic failover timing.
- `gatewayctl breaker force-close --upstream X` deletes the key. Same ~1s reaction time.
- Forgotten override expires naturally via Redis EX (max 300s enforced at the CLI layer).

**Read-overhead acceptance (WARNING-4 / Test 6 measurement):**
- 1000 `CheckForceOverride` calls with key ABSENT: <10ms total (≤10µs/call) — verified via `TestBreakerFSM_ForceOverrideReadOverheadUnderMs`.
- 1000 cached reads with key PRESENT: <50ms total (≤50µs/call) — same test, second half.
- Single first-call cost (cache miss): one Redis GET + JSON decode + PTTL, amortized across the 1s freshness window.

**Key namespace:**
- Force key: `gw:breaker:force:{upstream-name}` (prefix in `force_override.go:ForceOverrideKeyPrefix`; helper `ForceOverrideKey(name)`).
- Force value: JSON `ForceOverrideValue{State, TTLSec, SetBy, SetAt}` — Plan 04 ships open-only `State="open"`; `State` is string-typed for forward-compat.
- Plain breaker state mirror (existing): `gw:breaker:{upstream-name}` — unchanged by this seam. Force keys deliberately occupy a SUB-namespace so a `SCAN gw:breaker:force:*` enumerates only force keys.

**Cross-replica:**
The force-override Redis key is shared across all gateway replicas. Each replica polls the key independently (via its own `forceCache` + `maybeRefreshForceOverride`); within ~1s of an operator write, EVERY replica observes the force. No Pub/Sub event is published for force-override (unlike `remoteOpen` which uses the existing breaker Pub/Sub channel) — the key+TTL is simpler and matches the operator-action model (force is intentional + bounded; peer-observation is incidental + unbounded).

**Anti-patterns to avoid (for future contributors):**
- Do NOT make `Set.Execute` synchronously block on Redis when force-override is checked — the 1s freshness debounce is critical to keep ≤10µs/request p99.
- Do NOT cache force=false entries with infinite freshness — that would create a window where an operator force-open never takes effect (the cache would never refresh). The fixed 1s freshness applies to both set and unset entries.
- Do NOT publish a Pub/Sub event for force-override writes — operators want bounded-TTL keys with natural expiry, not append-only event streams.
