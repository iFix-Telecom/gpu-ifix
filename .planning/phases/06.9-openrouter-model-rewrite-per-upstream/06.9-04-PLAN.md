# R1 Scope Justification — Plan 04 stays bundled (not split into 04a/04b)

Both reviewers (Gemini + Codex) flagged Plan 04 as expanding from "model rewrite fix" into operator-tooling territory (breaker control + model-alias CRUD). Replan decision: **KEEP BUNDLED**.

**WARNING-6 (revision iteration 2) — Bundle partitioned into critical-path + operator-convenience tiers (still all in Plan 04, no deferral):**

| Tier | Subcommands | Why Bundled in Plan 04 | Could Be Deferred? |
|------|-------------|------------------------|--------------------|
| **Critical path — Plan 06 dependency** | `breaker force-open`, `breaker list`, `breaker force-close` | Plan 06 S2/S3 REQUIRE force-open (no real-failure driver for external upstreams without breaking OpenAI keys). force-close is required for cleanup-on-abort (R2 cleanup-on-abort mandate). list is required for verifying baseline + post-cleanup state. | NO — Plan 06 blocks without these. |
| **Operator-convenience — co-bundled with R7/R10** | `model-alias list`, `model-alias get`, `model-alias set`, `model-alias delete` | These DEPEND on Plan 01 R7's sqlc queries (UpsertModelAlias / DeleteModelAlias) AND Plan 04 R10's input validation. R7 + R10 land in Plan 01 + Plan 04 respectively; the CLI consumes both. Splitting CLI to a follow-up phase would orphan the R7 queries (they'd be added but unused) AND duplicate the test-infrastructure setup. | NO — co-bundled because the CLI is the natural consumer of R7 + R10 outputs. |

**Per D-06 + BLOCKER-1 (revision iteration 2):** `model-alias set` and `UPSTREAM_*_MODEL` env vars are COEQUAL operator override paths (not "old vs new"; both supported permanently). The CLI is the multi-instance-consistent path; env is the per-instance escape hatch.

**Trade-off acknowledged:** Plan 04 grows the diff surface by ~3 new files + 1 new Go package file (breaker/force_override.go). All new code is operator-only (gatewayctl is a CLI, not exposed to end users); deployment is access-restricted (only operators with shell access can run it). No authentication boundary lives INSIDE the CLI itself.

**R1 HIGH constraints applied (breaker forcing remains in scope WITH these guardrails):**
- **TTL mandatory** — `force-open` REJECTS calls without `--ttl <duration>`; max TTL = 300s (5min); breaker FSM honors TTL expiry via Redis SET EX (no manual cleanup needed if operator forgets force-close).
- **Audit log entry** on every force-open / force-close (reuses existing `audit_log` table per Plan 02 audit writer).
- **Explicit Redis key namespace** — `gw:breaker:force:{upstream-name}` documented in code + SUMMARY + Plan 06 cleanup + 06.9-PATTERNS.md (NEW patterns entry); value JSON `{state, ttl_sec, set_by, set_at}`.
- **Runtime read path (WARNING-4):** `gateway/internal/breaker/force_override.go` checks the force key on the existing breaker evaluation tick cadence; force takes PRECEDENCE over observation. Read cost MUST be ≤1ms per eval tick (single Redis GET, not per-request). Task 2 entry-gate confirms the canonical FSM read site + injection point BEFORE writing the patch.
- **Tests proving TTL expiry restores normal behavior** — Plan 05b integration test (R8 gate cleared) validates: force-open with TTL=5s → wait 6s → force key expired → breaker returns to observation-driven state.
- **Operator-only deployment note** — README + CLI `--help` text clarifies: "gatewayctl assumes operator-only access (Docker exec / shell). No auth boundary inside the CLI."
