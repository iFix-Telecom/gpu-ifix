<objective>
Make the existing `UsageInterceptor` WRITE the audio + embed usage atomics it already
reads at flush time. Today `usageJSONBuffer.Close` parses only the chat-token shape
(`usage.prompt_tokens` / `usage.completion_tokens`); STT and embed responses flow through
the SAME JSON-buffer path but their billable dimensions (audio duration, embed count) are
silently dropped — so `AudioSecondsMs10` / `EmbedsCount` are never produced. This is the
producer-less dangling-read at the heart of TEN-04 / OBS-09.

This plan adds the producers IN the existing interceptor (mirroring the chat-path token
extractor — locked decision #1: do NOT invent a new billing path) and proves the unit
conversion with tests. Wiring the 6 proxies to actually invoke this interceptor is Plan
16-02 (depends on this plan).

Purpose: turn the dead audio/embed metering dimension into live data feeding usage_counters.
Output: route-dispatched producers in interceptor_usage.go + unit tests locking the unit math.
</objective>
