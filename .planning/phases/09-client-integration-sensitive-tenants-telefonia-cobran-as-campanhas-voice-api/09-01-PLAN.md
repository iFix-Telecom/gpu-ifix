<objective>
Extend the Phase-8 idempotent seed script `scripts/integration-smoke/provision-tenants.sh` to provision the four Phase-9 client tenants — `telefonia` and `cobrancas` (`data_class: sensitive`), `campanhas` and `voice-api` (`data_class: normal`) — and apply per-tenant quotas to `cobrancas` + `campanhas` via `gatewayctl tenant set-quota`. This implements INT-03 (Telefonia sensitive), INT-04 (Cobranças + Campanhas), INT-05 (voice-api).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator shell → gatewayctl → gateway DB | The seed script runs with the gateway DB DSN and mints real tenant identities + API keys, two of which (`telefonia`, `cobrancas`) are `data_class: sensitive` and gate LGPD-relevant call-audio + financial data |
| script stdout / stderr → operator terminal → log files | Raw API keys (including the 2 sensitive-tenant keys) are emitted to stdout once; stderr (`log()`) may be redirected to a file by the operator |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Information Disclosure | the raw sensitive-tenant API keys (telefonia, cobrancas) printed by `provision-tenants.sh` | mitigate | Raw keys are captured into shell variables and emitted ONLY via the stdout final-instructions heredoc — never passed to `log()` (which writes to stderr and may be redirected to a file). Inherits the Phase-8 secret-once discipline (`parse_key` asserts exactly one `^key=` line). The heredoc warning states keys are shown once, must be copied straight into Portainer stack env vars, and must never be committed. |
| T-09-02 | Tampering | a `tenant create` succeeded but the matching `set-quota` silently failed, leaving `cobrancas`/`campanhas` quota-unbounded | mitigate | The quota step treats ANY non-zero `gatewayctl tenant set-quota` exit as fatal `exit 1` (NOT an idempotency-OK case — `set-quota` exits 1 with `tenant %q not found`, which can only mean tenant-create failed). `set -euo pipefail` is set. A half-provisioned tenant halts the script instead of silently shipping without a quota. |
| T-09-03 | Elevation of Privilege | a tenant key minted with the wrong `data_class` — e.g. `telefonia` keyed `normal` — would let LGPD-sensitive call audio fail over to OpenAI/OpenRouter | mitigate | The key-mint step passes the per-tenant `data_class` explicitly from the `TENANT_DATA_CLASS` array (`sensitive` for telefonia + cobrancas, `normal` for campanhas + voice-api). The data_class is the load-bearing input to the Phase-3 RES-08 sensitive-block path; getting it right at mint time is what makes the never-external guarantee hold. The Phase-9 sensitive-failover smoke (plan 09-02) is the black-box proof that the minted sensitive keys actually trigger the block. |
| T-09-04 | Elevation of Privilege | re-running the script sprays duplicate API keys, widening the auth surface | mitigate | `gatewayctl key create` / `admin-key create` are non-idempotent; both stay guarded behind the explicit `--mint-keys` opt-in. The default re-run path runs only the idempotent `tenant create` + `tenant set-quota` and logs the "re-run with --mint-keys exactly once" instruction. |
| T-09-05 | Repudiation | no record of which operator provisioned the sensitive tenants | accept | The `created_at` timestamps on `tenants` + `api_keys` (Phase 2 schema) capture provisioning provenance at the DB level; operator attribution lives in the Phase-9 HUMAN-UAT sign-off sheet (plan 09-04). No new mitigation needed in the seed script — same posture as Phase 8 (T-08-04). |
</threat_model>
