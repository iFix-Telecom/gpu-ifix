# Phase 9: Code Review Report

**Reviewed:** 2026-05-14T00:00:00Z
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Reviewed the Phase 9 sensitive-tenant integration-smoke trio: the extended
`provision-tenants.sh` tenant seed, the new RES-08 `smoke-sensitive-failover.py`
black-box smoke, and its report schema.

The threat-model focus areas hold up well: the bash `data_class`-per-tenant
array indexing is correct and parallel; the psycopg queries are fully
parameterized (no SQL injection surface); the DSN and tenant key are never
logged nor written to the report; secret-once discipline on key minting is
intact; the audit `COUNT(*)`-only query never pulls content columns.

However there is **one BLOCKER**: the smoke's central anti-leak assertion can
produce a **false GREEN** because `query_audit` treats `audit_upstream` as a
plain string compare against `blocked_sensitive` but never excludes the case
where the audit row was written with a *real external upstream* value AND the
`fail_closed` HTTP gate independently failed — combined with the streaming
gate firing a *second* request that overwrites correlation, the audit gate can
pass on the wrong request. More importantly, the `never_external` gate does
**not** require the `fail_closed` gate to have passed, so a request that was
actually proxied externally but happened to also create a `blocked_sensitive`
row for a *different* attempt can read GREEN. Details below.

## Critical Issues

### CR-01: `never_external` / `audit_decision` gates correlate on a request_id that may belong to a different request than the one proven to fail closed

**File:** `scripts/integration-smoke/smoke-sensitive-failover.py:606`, `:614`, `:626`, `:634-654`

**Issue:** The smoke fires up to **two** sensitive requests against the gateway
while tier-0 is open: `run_fail_closed_request` (line 605) then
`run_streaming_fail_fast_request` (line 614). The audit gates
(`never_external`, `audit_decision`) are correlated **only** by the
`request_id` captured from the *first* (non-streaming) request (line 606,
`request_id = fail_closed.pop("request_id", "")`).

Two concrete false-GREEN paths:

1. **`request_id` empty → audit gates degrade silently in a way that still
   permits a partial pass.** If the gateway response to the fail_closed request
   lacks an `X-Request-ID` header (e.g. the 503 is generated by an upstream
   proxy/edge layer before the gateway's requestid middleware runs, or the
   request errored), `request_id` is `""`. `query_audit` then returns
   `audit_log_row_found: False`, so `never_external_ok` is False and
   `audit["ok"]` is False — that part is fine. But the **`fail_closed` gate
   itself can still be True** (503 + envelope + Retry-After all present in the
   body even without the header). The operator sees `fail_closed: true` and is
   told the leak proof "couldn't correlate" — but there is no gate that asserts
   "a request_id MUST have been captured." A reviewer reading the report sees a
   GREEN fail_closed and an audit failure that *looks* like flaky DB access,
   not a missing-proof condition. The anti-leak proof is the whole point of
   this smoke; an un-correlatable run must be a hard failure, not a soft one.

2. **The `never_external` gate does not depend on `fail_closed` passing.** A
   request that *was* actually proxied to OpenAI/OpenRouter would produce an
   audit row with `upstream` = the real provider name, so `audit_upstream ==
   "blocked_sensitive"` would be False and the gate would fail — good in
   isolation. But because the two gates are evaluated **independently** and the
   exit-code contract treats them as separable (exit 3 = never_external only),
   there is no invariant enforced that `never_external` GREEN implies
   `fail_closed` GREEN. If a future change to `run_fail_closed_request` (or a
   gateway that returns 200 on a sensitive request — exactly the leak being
   guarded against) leaves `fail_closed.ok = False` but the audit row for that
   same request_id still happens to read `blocked_sensitive` (e.g. the gateway
   blocked-and-audited but a bug returned 200 to the client), the smoke reports
   `never_external: true`. The black-box proof of "never went external" is
   being asserted from the audit row alone while the HTTP-side proof is allowed
   to disagree.

**Fix:** Make the anti-leak proof conjunctive and require a correlatable
request_id as a hard precondition:

```python
# After Step 2, before evaluating audit gates:
if not request_id:
    msg = ("fail_closed request returned no X-Request-ID — the anti-leak "
           "audit proof cannot be correlated; refusing to evaluate audit "
           "gates (a non-correlatable run is a hard failure, not a soft one)")
    log.error("no request_id captured", detail=msg)
    errors.append(msg)
    _write_unevaluated_report(cfg, started_at, errors)
    return 1

# never_external must ALSO require the HTTP-side fail_closed proof:
never_external_ok = (
    fail_closed["ok"]                       # HTTP proof: 503 + envelope + Retry-After
    and audit["audit_log_row_found"]
    and audit["audit_upstream"] == AUDIT_UPSTREAM_BLOCKED
)
```

This guarantees a GREEN `never_external` can only happen when the request
provably failed closed at the HTTP layer *and* the audit row independently
confirms `blocked_sensitive` for that exact request_id.

## Warnings

### WR-01: `audit_decision` masks a negative `audit_log_content_rows` sentinel before writing the report

**File:** `scripts/integration-smoke/smoke-sensitive-failover.py:651`

**Issue:** `query_audit` initializes `audit_log_content_rows` to `-1` (line 444)
and on a DB error returns early leaving it `-1` (line 477-478). The schema
requires `audit_log_content_rows` to be `integer, minimum: 0` (schema line 74),
so line 651 does `max(audit["audit_log_content_rows"], 0)` to satisfy the
schema. The gate logic (`audit["ok"]`, line 480-484) correctly checks `== 0`
against the *raw* value so a `-1` does not pass the gate — that part is sound.
But the **written report** then shows `audit_log_content_rows: 0` even when the
DB query never ran. A human reading the report sees `audit_log_content_rows: 0`
which is the *exact value that means "no sensitive content persisted — PASS"* —
on a run where the query failed entirely. The `ok: false` + `raw_error_body`
fields are the only disambiguator, and they are easy to miss next to a `0` that
looks like a clean result.

**Fix:** Keep the on-error / not-run sentinel visible without breaking the
schema — either widen the schema to allow `-1` with a documented meaning, or
write the field only when the query actually ran and make it not-required, or
surface a distinct boolean:

```python
audit_decision: dict[str, Any] = {
    "status_code": fail_closed["status_code"],
    "ok": audit["ok"],
    "audit_log_row_found": audit["audit_log_row_found"],
    # -1 sentinel means "audit-DB query did not complete" — do NOT collapse
    # to 0, which is the PASS value. Schema updated to allow minimum: -1.
    "audit_log_content_rows": audit["audit_log_content_rows"],
}
```

### WR-02: streaming gate fires a second uncorrelated sensitive request against the same open breaker

**File:** `scripts/integration-smoke/smoke-sensitive-failover.py:610-622`

**Issue:** `run_streaming_fail_fast_request` issues a second POST
`/v1/chat/completions` with the sensitive tenant key. Its `request_id` is never
captured and never audited. If a regression made the streaming path *leak*
(proxy a sensitive streaming request externally), this smoke would not catch it
— only the `elapsed_ms < 500` + `503` timing/status check runs, with no
`never_external`/audit equivalent for the streaming request. The smoke's own
docstring claims it proves the streaming path fails fast per D-B4, but it does
not prove the streaming path *fails closed* the way the non-streaming path is
proven. This is a coverage gap on the highest-stakes (sensitive-data-leak)
threat, for the streaming code path specifically.

**Fix:** Capture the streaming request's `X-Request-ID` and run the same audit
correlation against it (`upstream == blocked_sensitive`, zero content rows), or
explicitly document in the docstring + schema that the streaming gate is a
timing-only check and the streaming leak path is covered by the in-process
`sensitive_block_test.go` only.

### WR-03: `--dry-run` skips the gatewayctl executable precheck but `set-quota` parallel arrays could still desync undetected

**File:** `scripts/integration-smoke/provision-tenants.sh:114-116`, `:169-173`

**Issue:** `QUOTA_TENANTS`, `QUOTA_DAILY_TOKENS`, `QUOTA_RPM` are three parallel
arrays indexed by `${!QUOTA_TENANTS[@]}` (line 169). If a future edit adds a
tenant to `QUOTA_TENANTS` but forgets one of the value arrays, `qtokens` or
`qrpm` silently becomes empty (`set -u` does *not* trip on an out-of-range
array index expansion — `"${QUOTA_RPM[$i]}"` for a missing index expands to
empty, not an error). `gatewayctl tenant set-quota --rpm ""` then runs with an
empty value. Same latent risk exists for `TENANT_SLUGS`/`TENANT_NAMES`/
`TENANT_DATA_CLASS` (lines 104-106) — the data_class array desync is explicitly
called out as a security bug in the review brief, and nothing in the script
asserts the three arrays are the same length.

**Fix:** Add a length-equality assertion immediately after the array
definitions:

```bash
[[ ${#TENANT_SLUGS[@]} -eq ${#TENANT_NAMES[@]} && \
   ${#TENANT_SLUGS[@]} -eq ${#TENANT_DATA_CLASS[@]} ]] \
  || { log "FATAL: tenant arrays desynced (slugs/names/data_class length mismatch)"; exit 1; }
[[ ${#QUOTA_TENANTS[@]} -eq ${#QUOTA_DAILY_TOKENS[@]} && \
   ${#QUOTA_TENANTS[@]} -eq ${#QUOTA_RPM[@]} ]] \
  || { log "FATAL: quota arrays desynced"; exit 1; }
```

### WR-04: `mint_tenant_key` data_class is passed positionally and is not cross-checked against `TENANT_DATA_CLASS`

**File:** `scripts/integration-smoke/provision-tenants.sh:104-106`, `:238-245`

**Issue:** The script defines `TENANT_DATA_CLASS=("sensitive" "sensitive"
"normal" "normal")` parallel to `TENANT_SLUGS` (lines 104-106) and the comment
at line 99-102 explicitly says that array "is consumed in the key-mint step
below." But the key-mint step (lines 238-245) does **not** consume it — it
hardcodes the data_class as a literal second argument to each
`mint_tenant_key` call (`mint_tenant_key telefonia sensitive`,
`mint_tenant_key campanhas normal`, ...). The `TENANT_DATA_CLASS` array is
therefore dead data, and the per-tenant data_class is now expressed twice (once
in the array, once inline) with nothing keeping them in sync. A future edit
that flips `cobrancas` to `normal` in the array but not in the inline call —
or vice versa — produces a tenant key with the wrong data_class, which the
review brief identifies as a security bug. The "single source of truth" the
array was meant to be is not actually the source of truth.

**Fix:** Either drive the key-mint loop from the arrays so there is one source
of truth, or delete the unused `TENANT_DATA_CLASS` array and update the
misleading comment:

```bash
for i in "${!TENANT_SLUGS[@]}"; do
  slug="${TENANT_SLUGS[$i]}"
  dc="${TENANT_DATA_CLASS[$i]}"
  key="$(mint_tenant_key "$slug" "$dc")"
  # ... assign into the right *_KEY var or an associative array
done
```

### WR-05: report-schema validation failure is swallowed — a malformed report still writes and the smoke still exits 0

**File:** `scripts/integration-smoke/smoke-sensitive-failover.py:689-707`

**Issue:** Schema validation is wrapped in `try/except Exception` that only
`log.warning`s on mismatch (lines 700-704), then the report is written and
`exit_code_for_gates` runs on `report["gates"]` regardless. The comment
justifies this as "schema drift should not crash a smoke that produced a real
report." But for a *security* smoke whose entire output contract is the report,
a schema-invalid report means a downstream consumer (the HUMAN-UAT asserter)
may misparse it — and the smoke can still exit `0` (all gates pass) while
emitting a structurally broken report. A false GREEN delivered via a
schema-invalid report is still a false GREEN. Also: `jsonschema` is imported
lazily inside the `try`, so if `jsonschema` is simply not installed, validation
is silently skipped entirely with only a warning — the smoke never validates
its own output and no one notices.

**Fix:** Distinguish "validator unavailable" from "report invalid." A genuinely
invalid report should force a non-zero exit (or at minimum a distinct
`errors[]` entry that the asserter keys on), and a missing `jsonschema`
dependency should be a hard startup error since it is listed in
`requirements.txt`:

```python
from jsonschema import Draft202012Validator  # import at module top; hard-fail if absent
...
try:
    Draft202012Validator(schema).validate(report)
except ValidationError as e:
    errors.append(f"report failed schema validation: {e.message}")
    # write the report for debugging, but ensure exit is non-zero
```

## Info

### IN-01: `print_operator_prestep` always prints even when the breaker is already open

**File:** `scripts/integration-smoke/smoke-sensitive-failover.py:578-585`

**Issue:** In `operator-prestep` mode the multi-line instruction block is
printed unconditionally (line 579) *before* `ensure_tier0_open` polls. If the
operator already tripped the breaker, the smoke confirms `open` on the first
poll but the operator was still shown a wall of "do this first" instructions —
mildly confusing. Cosmetic only.

**Fix:** Do a single fast probe first; only print the prestep block if
`local-llm` is not already `open`.

### IN-02: `git_sha` best-effort block silently swallows all exceptions

**File:** `scripts/integration-smoke/smoke-sensitive-failover.py:678-687`

**Issue:** `except Exception: pass` (lines 686-687) discards everything,
including non-git errors (e.g. `subprocess` not found, permission issues). The
`git_sha` field is genuinely optional per the schema so this is acceptable, but
a bare `pass` with no log line means a missing `git_sha` in the report is
indistinguishable from "git not present" vs "subprocess bug." Consider a
`log.debug` in the handler.

### IN-03: `GW_OUT` from a previous gatewayctl call leaks into the next iteration's failure log on a no-output command

**File:** `scripts/integration-smoke/provision-tenants.sh:121-134`

**Issue:** `GW_OUT` is a module-level var reused across every `run_gatewayctl`
call. `run_gatewayctl` always reassigns it (line 131), so in practice it is
fresh each call — but if a future edit adds a code path that checks `GW_OUT`
without calling `run_gatewayctl` first, it would read a stale value. Minor
robustness nit; current code is correct.

**Fix:** Reset `GW_OUT=""; GW_RC=0` at the top of `run_gatewayctl` before the
`set +e` block for defensiveness.

### IN-04: schema allows `retry_after` as string OR integer but the script only ever emits a string

**File:** `scripts/integration-smoke/sensitive-failover-report-schema.json:45`, `smoke-sensitive-failover.py:346`

**Issue:** Schema `fail_closed.retry_after` is `type: ["string", "integer"]`
(line 45) but `run_fail_closed_request` always assigns `r.headers.get(
"Retry-After", "")` — always a string. The integer branch of the union is dead
relative to this producer. Harmless, but the union is misleading to a schema
reader who might assume an integer form is produced somewhere. Tighten to
`type: string` or document why the union exists.
