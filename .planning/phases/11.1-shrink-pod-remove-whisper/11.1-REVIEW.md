# Phase 11.1: Code Review Report

**Reviewed:** 2026-06-05
**Depth:** standard
**Files Reviewed:** 22
**Status:** issues_found

## Summary

A fase 11.1 entrega a remoção de STT tier-0 (Speaches/Whisper) do pod e do
schema do gateway. Os diffs estão internamente coerentes (config split em
primary/fallback, reconciler iterando shapes, health-bridge removendo
:8001/UpstreamSTT, supervisord enxuto, migration 0028 com Down idempotente),
**porém o trabalho está incompleto fora do diff explícito**: o gateway main
ainda exige `UPSTREAM_STT_URL` e instancia `audioRP` contra ele; o onstart
embutido em `gateway/internal/primary/onstart.go` continua exigindo e
baixando weights de Whisper (`PRIMARY_WHISPER_WEIGHTS_KEY/SHA256`); o
`pod/scripts/download-weights.sh` (encadeado por `pod/onstart.sh`) também
exige `WEIGHTS_WHISPER_KEY/SHA256`. Resultado: gateway **não bootará** em
prod após drop do env, pod primário **falhará no provision** (build request
rejeitada por `ErrMissingWhisperSHA`) e pod template **falhará no
download-weights**. Há também 3 referências stale ao shape antigo (5090
$2.20, 4090) em comments e ao número de migration (0029 vs 0028).

## Critical Issues

### CR-01: pod/onstart.sh removeu env vars Whisper mas script invocado ainda exige

**File:** `pod/onstart.sh:46-56` (em conjunto com `pod/scripts/download-weights.sh:27`)
**Issue:** O diff de `pod/onstart.sh` removeu `: "${WEIGHTS_WHISPER_KEY:?}" : "${WEIGHTS_WHISPER_SHA256:?}"` das guards iniciais. Mas o `download-weights.sh` invocado em seguida (resolvido em `SCRIPT_DIR/scripts/` ou `IFIX_AI_POD_ROOT/scripts/`) ainda contém na linha 27:
```bash
: "${WEIGHTS_WHISPER_KEY:?missing}" "${WEIGHTS_WHISPER_SHA256:?missing}"
```
Com `set -euo pipefail` o pod aborta com exit ~1 e nunca chega a `docker compose up`. Toda a redução de imagem fica inacessível porque o pod nem inicia.
**Fix:** Aplicar a mesma poda em `pod/scripts/download-weights.sh` (remover linhas 27, 44 `${WEIGHTS_DIR}/whisper`, 74 `WHISPER_DEST`, 79 download, 98-99 extract, 109 ls). O escopo do plano 11.1-05 (env hygiene) deveria ter incluído esse arquivo — adicionar como Task de hot-fix.

### CR-02: gateway/cmd/gateway/main.go continua exigindo UPSTREAM_STT_URL + criando audioRP

**File:** `gateway/internal/config/config.go:481-488`, `gateway/cmd/gateway/main.go:570`
**Issue:** `requiredOrder` ainda contém `"UPSTREAM_STT_URL"` e o map `required` ainda lê `cfg.UpstreamSTTURL`. Em `main.go:570`, `proxy.NewAudioProxy(cfg.UpstreamSTTURL, log)` é instanciado e wired como `"local-stt": audioRP` em `sttRoleProxies` (linha 661). O RUNBOOK-DEPLOY.md (T-11.1-02) instrui o operador a **deletar** `UPSTREAM_STT_URL` do `.env` prod e dar `docker compose up -d`. Resultado: o container não sobe — o Load() retorna `ErrMissingEnv: UPSTREAM_STT_URL` e o gateway crasha em loop. A documentação e o código estão em conflito direto: o runbook torna o ambiente inconsistente, exatamente o oposto da hygiene pretendida.
**Fix:** Antes do PR mergeable, remover do `requiredOrder` + `required` map, remover `UpstreamSTTURL` da struct (ou mantê-la como deprecated-warn com aviso semelhante ao alias D-A6), remover instanciação de `audioRP` e a entrada `"local-stt"` em `sttRoleProxies`. Alternativamente, downgrade da var para opcional (`envOr` default ""), com o wiring de `local-stt` condicional em `cfg.UpstreamSTTURL != ""` — mas isso é apenas migração transitória; o end state desejado per D-A4 é remoção total.

### CR-03: Primary reconciler ainda exige PRIMARY_WHISPER_WEIGHTS_SHA256 + onstart.go embutido baixa whisper

**File:** `gateway/internal/primary/lifecycle.go:302-307`, `gateway/internal/primary/onstart.go:114-115,145-146,156,159`
**Issue:** `buildCreateRequest` rejeita o request com `ErrMissingWhisperSHA` quando `cfg.PrimaryWhisperWeightsSHA256 == ""`. O onstart embutido em `gateway/internal/primary/onstart.go` (string `primaryOnstartHead`) também tem:
```bash
: "${PRIMARY_WHISPER_WEIGHTS_KEY:?required}"
: "${PRIMARY_WHISPER_WEIGHTS_SHA256:?required}"
…
download_with_verify "$PRIMARY_WHISPER_WEIGHTS_KEY" "/weights/whisper/model.tar.gz" "$PRIMARY_WHISPER_WEIGHTS_SHA256" &
…
tar -xzf /weights/whisper/model.tar.gz -C /weights/whisper
```
Com a fase 11.1 D-A4 instruindo operadores a deixar de setar `PRIMARY_WHISPER_WEIGHTS_*`, a primeira tentativa do reconciler de provisionar pod cai em Cooldown perpétuo (sentinel error). Mesmo se operador setar a SHA antiga por inércia, o `wait $QWEN_PID $WHISPER_PID $BGE_PID` baixa ~3 GB extras inúteis e o `tar -xzf` extrai em diretório que o supervisord nem usa mais.
**Fix:** Remover os 3 campos `PrimaryWhisperWeightsKey/SHA256/...` de `config.go` (e do log `Load()` 2 ocorrências) + `ErrMissingWhisperSHA` + branch em `buildCreateRequest` + 3 entradas de env no map (linhas 345-346) + as 3 ocorrências de whisper no `primaryOnstartHead` (download + wait + tar). Atualizar `primary/onstart_test.go` para refletir.

## Warnings

### WR-01: pod/primary/Dockerfile mantém /weights/bge-m3 mkdir mas BGE-M3 saiu do pod (D-03)

**Issue:** O mkdir cria `/weights/qwen /weights/bge-m3 /app/templates /opt/chatterbox-data/...`. Mas D-03 (Phase 06.7) relocou embed para o host CPU; D-A2/A4 (Phase 11.1) drop STT. Sobra apenas `qwen` como weights real. `bge-m3` é orphan mkdir.
**Fix:** Remover `/weights/bge-m3` do mkdir. Verificar se algum env / volume / script de upload ainda referencia esse diretório no pod primário.

### WR-02: pod/.env.example menciona "bge-m3/ (HF cache)" como obrigatório mas embed também é off-pod

**File:** `pod/.env.example:13`
**Issue:** Comment "Must contain: qwen/model.gguf, bge-m3/ (HF cache)" — mas BGE-M3 saiu do pod desde D-03 (Phase 06.7). O `pod/docker-compose.yml` removeu o serviço `infinity` na Phase 11.1, então não existe consumidor de `bge-m3/` no pod template. Operadores serão induzidos a baixar weights inúteis.
**Fix:** Reduzir comment a "Must contain: qwen/model.gguf". Considerar dropar também `WEIGHTS_BGE_M3_KEY` + `WEIGHTS_BGE_M3_SHA256` do template `.env.example` (linhas 31, 34) já que o serviço infinity não está mais no compose.

### WR-03: Comentários de PrimaryVastPriceCapDPH/PrimaryHostID ainda descrevem 5090 EU $2.20 — shape efetivo agora é 3090

**File:** `gateway/internal/config/config.go:194`
**Issue:** O comment do campo `PrimaryVastPriceCapDPH` (DEPRECATED alias) descreve "default 2.20; RTX 5090 EU cap — UAT 17 2026-05-19 picked 5090 32 GB to fit Qwen 27B + bge-m3 + KV cache + whisper-large-v3 GPU…" — referenciando workload de bge-m3 + whisper que agora estão removidos. O default real lido em `floatOr(...,0.30)` está só nos novos campos, mas o comment legado dá impressão errada de invariante.
**Fix:** Anotar no comment do legacy `PrimaryVastPriceCapDPH` que o cap histórico era 2.20 PORÉM Phase 11.1 D-A6 reduziu para 0.30 (primary) / 0.60 (fallback) com 1-2×3090. Idem para `PrimaryGPUName` ("RTX 5090") e `PrimaryNumGPUs` (default 1) — quando legacy var é consumida ela sobreescreve PrimaryVastGPUNamePrimary; se o operador deixou `PRIMARY_GPU_NAME=RTX 5090` em `.env` antigo, o reconciler vai procurar 5090 em Vast (acima do cap 0.30) e nunca achar offers — sintoma silencioso de "Primary nunca sobe".

### WR-04: pickedShape descartado com `_ = pickedShape` — perda de observabilidade

**File:** `gateway/internal/primary/reconciler.go:824`
**Issue:** `var pickedShape int` é setado dentro do loop mas no fim:
```go
_ = pickedShape // logged above; reserved for future per-shape metrics.
```
O log dentro do loop (`"primary offers found for shape"`) só dispara quando há match; mas o `pickedShape` que sobreviveu fora do loop **não é propagado** ao DB (audit `health_pass`) nem ao `ClosePrimaryLifecycle` event. Sem propagação, fica impossível diferenciar em produção se a primary subiu no shape 0 (cheap path) ou 1 (fallback 2×3090) — exatamente o sinal mais importante para decidir se a topologia D-A6 está economizando. Comment promete "reserved for future per-shape metrics" mas a métrica é necessária no MERGE deste plano, não no próximo.
**Fix:** Persistir `pickedShape` no event `health_pass` (`urls.LLM, ..., shape: pickedShape`) ou em label de métrica `gateway_primary_shape{shape="0"}`. Mínimo: propagar via `log.Info("primary picked offer", "shape", pickedShape, "offer_id", offer.ID)` antes do `offer := pickable[0]`.

### WR-05: allowlist preference aplicada APENAS ao shape primary — fallback shape perde otimização

**File:** `gateway/internal/primary/reconciler.go:792-805`
**Issue:** O bloco `if len(r.cfg.PrimaryVastMachineAllowlist) > 0` aplica `WithMachineAllowlist` somente em `filters[0]`. Quando allowlist tem máquinas 2×3090 (e.g. machine_id 43803 e 55158 — citados em primary-gpu-shape-06.8-final memory), o allowlist-first NUNCA testa o fallback shape. Se o operador catalogou hosts 2×3090 conhecidos-bons no allowlist, eles serão pulados na preferência inicial e só achados na broadening pass do shape 1, sem prioridade.
**Fix:** Decidir explicitamente o design — ou (a) iterar `for i, f := range filters { allowFilter := WithMachineAllowlist(f, allowlist) ... }` para ambas as shapes, ou (b) documentar no comment que allowlist é shape-0-only e renomear `PRIMARY_VAST_MACHINE_ALLOWLIST` → `PRIMARY_VAST_MACHINE_ALLOWLIST_PRIMARY` com um `_FALLBACK` separado. Memory `primary-gpu-shape-06.8-final` lista 2×3090 IDs no allowlist atual, sugerindo o caso (a) é o esperado.

### WR-06: RUNBOOK-DEPLOY.md cita "migração 0029" mas o arquivo é 0028
