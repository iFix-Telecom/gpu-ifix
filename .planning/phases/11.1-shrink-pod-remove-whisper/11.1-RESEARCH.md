# Phase 11.1: shrink-pod-remove-whisper — Research

**Researched:** 2026-06-04
**Domain:** Pod template refactor + Go-gateway DB-schema decommission + Vast.ai search-filter dispatch refactor
**Confidence:** HIGH (grep-verified against current tree on `gsd/phase-06.9-close` branch)

## Summary

This is a **subtractive** refactor with three orthogonal touchpoint axes:

1. **Pod image / runtime** — drop Stage 1b speaches-assets, `/opt/speaches-venv`, `[program:speaches]`, whisper weight bootstrap, `:8001` probe, and (bundled) `/opt/infinity-venv` + onstart embed log line.
2. **Gateway DB + Go runtime** — DELETE migration for `upstreams.local-stt` + `model_aliases (whisper, local-stt)`. Audit the gateway's role-`stt` code path so a missing tier-0 row resolves cleanly to `openai-whisper` instead of NPE/breaker-OPEN.
3. **Vast.ai DefaultSearchFilter** — extend `gateway/internal/emerg/vast/types.go` + `gateway/internal/config/config.go` with a primary/fallback GPU-shape dispatch (1×3090 cap $0.30 → 1×4090 cap $0.40).

**The single largest risk** is **NOT the SQL DELETE** (graceful — the gateway never crashes on missing rows) but a **set of hard-coded `"local-stt"` string references in live Go runtime paths**: `gateway/internal/primary/reconciler.go:448` (`Inflight.Count("local-stt")` summed into drain gate), the `roleURL` STT branch (`lifecycle.go:448`), and `markReady`'s `OverrideTier0("stt", urls.STT)` call (`reconciler.go:573`). After D-A4, `urls.STT` becomes meaningless and `OverrideTier0("stt", ...)` overrides a tier-0 slot that no longer exists in the upstreams table. The runtime won't crash (loader is map-keyed), but stale STT override URLs will silently route nowhere; the drain gate's `Count("local-stt")` always returns 0 (defensible — see `inflight.go:166`). **Decision needed at plan time:** strip the role=stt dynamic-primary path entirely (clean) vs. leave dead branches (minimal-diff). Recommend strip — Phase 06.7 D-03 already established the precedent ("embed is NOT a dynamic primary role").

**Primary recommendation:** Sequence the work as **(1) reconciler/lifecycle code-only PR removing the `stt` role from the dynamic-primary trio → tests green → (2) DB migration 0028 DELETE rows → (3) pod Dockerfile/supervisord/onstart subtraction + Infinity venv removal → (4) Vast search-filter primary/fallback refactor → (5) docs/runbook update + smoke test prune**. The reconciler change MUST land before the migration to avoid a window where a redeploy of stale gateway code dereferences a deleted row.

### Pattern 1: Migration Down = Re-seed (canonical, per `0026` precedent)
**What:** Goose migration ships symmetric Up/Down. Up = DELETE rows; Down = INSERT-with-ON-CONFLICT-DO-NOTHING to restore the Phase 06.9 baseline shape.
**When to use:** Whenever DELETE removes seed data that earlier migrations established.
**Example (from `0026_evolve_model_aliases_per_upstream.sql:81+`):** Down explicitly DELETEs the Up's INSERTs + restores prior PK. For 11.1, Down should re-INSERT `('local-stt', 'stt', 0, 'UPSTREAM_STT_URL', NULL)` + `('whisper', 'stt', 'Systran/faster-whisper-large-v3', 'local-stt')`.

### Pattern 2: Vast Filter Variadic Fallback
**What:** Compose two `SearchFilter` maps; reconciler tries primary, on empty result tries fallback before backing off.
**When to use:** GPU shape scarcity hedging.
**Example skeleton (proposed for `gateway/internal/emerg/vast/types.go`):**
```go
// DefaultSearchFilters returns the canonical primary + fallback filter pair.
// Reconciler tries [0] first; on empty offers tries [1]; on both-empty backs off.
func DefaultSearchFilters(primaryCap, fallbackCap float64, primaryHostID int64,
    primaryGPU, fallbackGPU string, numGPUs int, blocklist ...int64) []SearchFilter {
    return []SearchFilter{
        DefaultSearchFilter(primaryCap, primaryHostID, primaryGPU, numGPUs, blocklist...),
        DefaultSearchFilter(fallbackCap, primaryHostID, fallbackGPU, numGPUs, blocklist...),
    }
}
```
Reconciler call site changes from one `SearchOffers` call to a small loop over the slice.

### Anti-Patterns to Avoid
- **Leaving `urls.STT` as `string` in `primaryPodURLs` with empty value:** silent bug. Either remove field or set sentinel. Recommend remove (Go zero-value `""` still flows through `OverrideTier0` and corrupts state).
- **Soft-disable via `enabled=false`:** explicitly rejected by D-A4. Don't propose as alternative.
- **Patching `model_alias.go` hardcoded map after migration:** the map is read by `gatewayctl model-alias set` validation. Forgetting to delete `"local-stt": "stt"` lets operators re-insert a row that targets a nonexistent upstream.

## Common Pitfalls

### Pitfall 1: Reconciler drain-gate goes stale after row delete (HIGH severity)
**What goes wrong:** `reconciler.go:448` sums `Count("local-llm") + Count("local-stt") + Count("local-embed")` for the drain gate. After 11.1, the `"local-stt"` map key never gets written by dispatcher (no upstream row → no inflight tracking under that name), so `Count("local-stt")` always returns 0. Sum still computes correctly (0 contribution). **NOT a runtime bug** but a dead branch.
**Why it happens:** Tier-0 role trio assumption from Phase 06.6 Plan 06a is hardcoded.
**How to avoid:** In the reconciler refactor task, drop the `local-stt` term and update the surrounding doc comment (and `lifecycle.go:65`, `inflight.go:172` comments) to reflect the new 2-role trio (`local-llm + local-embed`).
**Warning signs:** Grep for `local-stt"` after the refactor — should return 0 hits in `gateway/internal/primary/` and `gateway/internal/shed/`.

### Pitfall 2: `OverrideTier0("stt", ...)` after row delete creates phantom override (MEDIUM)
**What goes wrong:** `reconciler.go:573` calls `Loader.OverrideTier0("stt", stripPrimaryReadinessSuffix(urls.STT))` after pod becomes ready. `Loader.OverrideTier0` (loader.go:250) writes to a role→URL map; resolution consults this BEFORE the upstreams DB row. If the override survives but the row is gone, requests with `role=stt` would hit the phantom URL.
**Why it happens:** Phase 06.7 D-03 removed `embed` from this path but stt + tts remain.
**How to avoid:** In the lifecycle/reconciler refactor, remove `urls.STT` field from `primaryPodURLs` (lifecycle.go:103), the `stt` case in `roleURL` (lifecycle.go:448), the `OverrideTier0("stt", ...)` call (reconciler.go:573), and the `RestoreTier0("stt")` calls (reconciler.go:525). Pattern mirror: how `embed` was removed in Phase 06.7 D-03.
**Warning signs:** Resolver miss test (`openai_whisper_model_rewrite_test.go` `Test_PassThrough...`) must continue green AFTER removal of the local-stt seed and driveBreaker setup.

### Pitfall 3: Migration 0028 cascade ordering (LOW with mitigation)
**What goes wrong:** If 0028 DELETEs `upstreams.local-stt` before DELETing `model_aliases (whisper, local-stt)`, FK constraint (if any from 0026 PK change `(alias, upstream_name)` — there's no FK to upstreams, just text) may complain. Verified: 0026's `model_aliases.upstream_name` is plain TEXT with a COMMENT, NOT a foreign key reference.
**Why it happens:** Reading 0026 carefully — line 75 confirms text-only contract.
**How to avoid:** No FK exists, so order is arbitrary. Recommended order anyway: DELETE model_aliases row first (more specific), then upstreams row (broader). Down inserts in reverse.
**Warning signs:** Local `goose up`/`goose down`/`goose up` roundtrip on a clean DB must leave schema identical.

### Pitfall 4: Phase 06.9 `(alias, upstream_name)` PK still valid post-delete (LOW)
**What goes wrong:** After deleting `(whisper, local-stt)`, the only remaining `whisper` row is `(whisper, openai-whisper)`. Resolver `Resolve("whisper", "openai-whisper")` continues to return `"whisper-1"`. **But** if a test seeds a row with `upstream_name=''` or different, behavior diverges.
**Verified:** `openai_whisper_model_rewrite_test.go:25` already documents the "no (whisper, openai-whisper) row" pass-through case — production has the row, so resolver returns `whisper-1`. Test currently calls `driveBreaker(t, bs, "local-stt", 500, 3)` to force tier-1 routing — post-11.1, the dispatcher has only one upstream registered for role=stt, so the breaker drive is unnecessary. Test fixture refactor required.
**Warning signs:** `openai_whisper_model_rewrite_test.go:198-219` setup harness needs simplification.

### Pitfall 5: `inet_down` 200 Mbps + 16GB weights still violates Phase 6 SC-2 P90 ≤6min (KNOWN, track-not-block)
**What goes wrong:** Phase 06.6 measured cold-start 20m58s with 16.74GB WAN download at the 200 Mbps inet floor. Removing 5GB whisper-large-v3 → ~14-15min projected. Phase 6 SC-2 (P90 cold-start ≤6min) remains unmet.
**Why it happens:** Hetzner/EU 3090 inventory inet ceiling caps download throughput.
**How to avoid:** Per CONTEXT.md "Claude's Discretion", explicitly mark SC-2 re-baselining as deferred-not-blocking; capture actual cold-start as retro evidence in `11.1-EVIDENCE.md`. Real solution = image pre-bake (Phase 12+ deferred).
**Warning signs:** First cold-start post-deploy >15min → investigate; <10min → mark SC-2 within target.

### Pitfall 6: Vast 4090 fallback inet_down ceiling unverified (MEDIUM, verify before plan)
**What goes wrong:** `DefaultSearchFilter` lowered inet_down from 500→200 specifically for the 2×3090 EU/DE/PL fleet (`types.go:195`). 4090 EU fleet may have different inet distribution — if it sits below 200 Mbps the fallback query returns 0 offers reliably.
**Why it happens:** 4090 inventory not surveyed during Phase 06.6 inet recalibration.
**How to avoid:** Plan should include a `checkpoint:human-verify` task: run `pod/scripts/vast-ai.sh search --gpu "RTX 4090"` and confirm >=3 qualified offers in EU/DE/PL under cap=$0.40/h. If fewer, drop inet_down for 4090 fallback to 100 Mbps OR accept smaller search radius.
**Warning signs:** Plan must NOT assume 4090 fallback availability without verification.

